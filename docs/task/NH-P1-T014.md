## 6. Business Rules

Implement these exact rules:

1. The candidate owner can always access the owner preview while authenticated.
2. External users can access only what the privacy service allows.
3. `PRIVATE` profiles are never externally accessible.
4. `LINK_ONLY` profiles require a valid opaque share token.
5. `PLATFORM_DISCOVERABLE` profiles may be accessed through the approved public identifier.
6. Section-level privacy settings override overall discoverability for individual sections.
7. Hidden sections must be omitted from API responses, not merely hidden in the frontend.
8. Sensitive identity and authentication fields must never be included.
9. Email, phone, date of birth, precise IP, session, audit, and token data must remain private.
10. Internal database IDs should be omitted where not required.
11. The external route must return safe not-found behavior for private, disabled, invalid-token, and nonexistent profiles.
12. The owner preview may show which sections are hidden, but must not weaken privacy.
13. Share tokens must be cryptographically random and stored only as hashes.
14. Raw share tokens must never be logged or audited.
15. Share-token rotation invalidates previous links.
16. Share-link disabling invalidates external link-only access.
17. Public professional URLs must use the URL-safety rules from `NH-P1-T012`.
18. Public profile data must come from authoritative backend filtering.
19. Profile-completion score must not be exposed externally unless privacy requirements explicitly permit it; default: owner-only.
20. Suspended or deleted accounts must not be externally visible.
21. All timestamps use UTC.
22. API responses must not leak hidden-section existence through detailed errors.
23. Audit metadata must not contain profile content or share tokens.
24. A blocked task must not create a Git commit.
25. A Git commit occurs only after all mandatory checks pass.

## 7. Preview Modes

Support these logical modes:

```text
OWNER
LINK_ONLY
PLATFORM_DISCOVERABLE
```

### OWNER

- authenticated candidate only
- returns visible profile plus owner-only privacy summary
- may include hidden-section placeholders
- may include completion percentage
- must never include auth/session secrets

### LINK_ONLY

- external access through valid share token
- available only when overall visibility is `LINK_ONLY`
- returns only externally allowed sections
- no owner-only metadata
- invalid/rotated/disabled token returns safe not-found

### PLATFORM_DISCOVERABLE

- external access through approved stable candidate public identifier
- available only when overall visibility is `PLATFORM_DISCOVERABLE`
- returns only externally allowed sections
- no owner-only metadata

## 9. Share Token Design

If `LINK_ONLY` requires a new token model, implement the logical equivalent of:

```prisma
model CandidateProfileShareToken {
  id         String    @id @default(uuid()) @db.Uuid
  userId     String    @unique @db.Uuid
  tokenHash  String    @unique @db.VarChar(128)
  enabled    Boolean   @default(true)
  rotatedAt  DateTime?
  createdAt  DateTime  @default(now())
  updatedAt  DateTime  @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([enabled])
}
```

Requirements:

- raw token uses cryptographically secure random bytes;
- raw token is shown only when created or rotated;
- only token hash is stored;
- token is at least 32 random bytes;
- rotation replaces the hash atomically;
- disabling sets access unavailable;
- token never appears in logs, audit metadata, or database plaintext.

If an equivalent secure sharing token already exists, reuse it instead of creating a duplicate model.
