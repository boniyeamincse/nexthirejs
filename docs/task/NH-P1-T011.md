## 6. Business Rules

Implement these exact rules:

1. Only authenticated users with the `candidate` role can use this feature.
2. A candidate can access only their own certification and training records.
3. Maximum 30 certifications per candidate.
4. Maximum 30 training records per candidate.
5. Certification name is required.
6. Certification issuer is required.
7. Certification issue date is required.
8. Certification expiry date is optional.
9. Expiry date cannot be earlier than issue date.
10. `doesNotExpire` and `expiryDate` must be logically consistent.
11. Credential ID is optional.
12. Credential URL is optional and must be a valid HTTP/HTTPS URL.
13. Training title is required.
14. Training provider is required.
15. Training completion date is required.
16. Training duration is optional.
17. Training description is optional and length-limited.
18. Ordering is candidate-controlled.
19. Reorder input must contain only owned record IDs.
20. Duplicate records are not silently merged.
21. Empty optional strings normalize to `null`.
22. Profile completion is recalculated after create/update/delete.
23. Backend completion is authoritative.
24. Audit metadata must not include certification names, provider names, credential IDs, URLs, or descriptions.
25. API authorization is authoritative.
26. Suspended or deleted accounts cannot modify records.
27. Cross-user access returns controlled not-found/forbidden behavior.
28. Request bodies must not accept arbitrary user IDs.
29. All timestamps use UTC.
30. Git commit occurs only after all mandatory checks pass.
31. A blocked task must not create a commit.

## 7. Certification Model

Implement:

```ts
{
  name: string;
  issuer: string;
  issueDate: string;
  expiryDate?: string | null;
  doesNotExpire: boolean;
  credentialId?: string | null;
  credentialUrl?: string | null;
}
```

### Validation

- `name`: required, trimmed, 2–200 characters.
- `issuer`: required, trimmed, 2–200 characters.
- `issueDate`: required ISO date and not in the future.
- `expiryDate`: optional ISO date, not before issue date, and null when `doesNotExpire` is true.
- `credentialId`: optional, trimmed, maximum 150 characters.
- `credentialUrl`: optional valid HTTP/HTTPS URL, maximum 500 characters.

## 9. Database Design

Inspect existing Prisma conventions.

Implement the logical equivalent of:

```prisma
model CandidateCertification {
  id            String    @id @default(uuid()) @db.Uuid
  userId        String    @db.Uuid
  name          String    @db.VarChar(200)
  issuer        String    @db.VarChar(200)
  issueDate     DateTime  @db.Date
  expiryDate    DateTime? @db.Date
  doesNotExpire Boolean   @default(false)
  credentialId  String?   @db.VarChar(150)
  credentialUrl String?   @db.VarChar(500)
  sortOrder     Int       @default(0)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, sortOrder])
  @@index([userId, issueDate])
}

model CandidateTraining {
  id             String    @id @default(uuid()) @db.Uuid
  userId         String    @db.Uuid
  title          String    @db.VarChar(200)
  provider       String    @db.VarChar(200)
  completionDate DateTime  @db.Date
  durationHours  Decimal?  @db.Decimal(7,1)
  description    String?   @db.Text
  sortOrder      Int       @default(0)
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, sortOrder])
  @@index([userId, completionDate])
}
```

If the repository avoids `Decimal`, store duration as validated integer minutes and document the decision.

### Ordering

- Assign next sort order on create.
- List ascending by sort order, then latest date.
- Reorder atomically.
- Normalize order after delete when practical.

### Migration safety

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not run `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
- Document migration results.

### Training

```http
GET    /api/v1/candidates/me/training

PUT    /api/v1/candidates/me/training/:trainingId
DELETE /api/v1/candidates/me/training/:trainingId
PUT    /api/v1/candidates/me/training/reorder
```

### Success statuses

- list: `200 OK`
- create: `201 Created`
- update: `200 OK`
- delete: `204 No Content`
- reorder: `200 OK`

```text
400 CANDIDATE_CERTIFICATION_VALIDATION_FAILED
```
