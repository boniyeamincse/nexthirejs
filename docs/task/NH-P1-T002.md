## 5. Business Rules

Implement these exact rules:

1. Only candidate accounts created through the existing registration flow are covered by this task.
2. A verification token is single-use.
3. The raw token must never be stored in the database.
4. Store only a cryptographic hash of the token.
5. Raw tokens must not be logged or included in audit metadata.
6. Tokens expire after a configurable duration.
7. Default development expiry: 24 hours.
8. A successful verification sets `emailVerifiedAt`.
9. A successful verification changes status from `PENDING_VERIFICATION` to `ACTIVE`.
10. Verification must be transactional.
11. Expired, invalid, consumed, or revoked tokens must not activate the account.
12. Repeated verification of an already verified account returns a safe idempotent result.
13. Resend must not reveal whether an email address belongs to an account.
14. Resend must revoke or invalidate previous unused tokens for the same user.
15. Resend must be rate limited.
16. Verification attempts must be rate limited.
17. Email delivery must happen asynchronously through the existing BullMQ foundation.
18. A failed email job must use the existing retry/dead-letter strategy.
19. A registration response must not fail solely because email delivery is asynchronous.
20. Email verification links must use the configured web base URL.
21. No production URL or secret may be hardcoded.
22. Audit records must exclude raw email addresses and tokens unless the existing privacy policy explicitly permits hashed identifiers.
23. Errors must not leak token hashes, database details, queue internals, or stack traces.
24. Timestamps must use UTC.
25. The frontend must not store the verification token in local storage, session storage, cookies, analytics, or persistent state.

## 6. Token Design

Use a cryptographically secure random token.

Recommended logical design:

```text
raw token: 32 or more random bytes
transport: base64url or hex
stored value: SHA-256 or stronger one-way hash of raw token
```

Requirements:

- use Node's cryptographic random generator;
- use a URL-safe representation;
- store only the hash;
- compare hashes safely;
- make tokens sufficiently unpredictable;
- never derive the token only from user ID, email, timestamp, or UUID;
- never use plain UUID as the full verification secret.

Suggested environment variables:

```text
EMAIL_VERIFICATION_TOKEN_TTL_MINUTES=1440
WEB_APP_URL=http://localhost:3000
EMAIL_FROM_ADDRESS=no-reply@nexthire.local
EMAIL_FROM_NAME=NextHire
```

Use existing variable names when already defined.

```prisma
model EmailVerificationToken {
  id          String    @id @default(uuid()) @db.Uuid
  userId      String    @db.Uuid

  expiresAt   DateTime
  consumedAt  DateTime?
```

## 10.1 Verify Email

### Endpoint

```http
POST /api/v1/auth/email-verification/verify
```

### Access

Public.

### Request

```json
{
  "token": "url-safe-raw-token"
}
```

### Success

```http
200 OK
```

Logical data:

```json
{
  "verified": true,
  "status": "ACTIVE",
  "emailVerifiedAt": "2026-07-21T10:00:00.000Z"
}
```

Use the repository-standard response envelope.

### Controlled errors

```text
400 AUTH_EMAIL_VERIFICATION_INVALID
400 AUTH_EMAIL_VERIFICATION_EXPIRED
400 AUTH_EMAIL_VERIFICATION_CONSUMED
429 AUTH_EMAIL_VERIFICATION_RATE_LIMITED
500 INTERNAL_SERVER_ERROR
```

For an already verified user, prefer an idempotent safe success response:

```json
{
  "verified": true,
  "alreadyVerified": true,
  "status": "ACTIVE"
}
```

Do not reveal private account details.

## 10.2 Resend Verification

### Endpoint

```http
POST /api/v1/auth/email-verification/resend
```

### Access

Public.

### Request

```json
{
  "email": "candidate@example.com"
}
```

### Success

Always return a generic accepted result:

```http
202 Accepted
```

Logical data:

```json
{
  "accepted": true,
  "message": "If an eligible account exists, a verification email will be sent."
}
```

Use the same response for:

- unknown email;
- already verified account;
- pending candidate account;
- ineligible account.

Internally, only eligible pending accounts receive a new token/email.

### Rate limit

Recommended:

```text
3 resend requests per email/IP per 15 minutes
```

Use the existing limiter mechanism.

### Swagger

Document both endpoints, response behavior, generic resend semantics, and security constraints.
