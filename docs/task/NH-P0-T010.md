## 3. Why the Project Must Be Run Now

Configuration files alone do not prove that the project works.

The project must be run because runtime verification can reveal:

- Invalid environment variables
- Port conflicts
- Docker health-check failures
- PostgreSQL connection errors
- Redis connection errors
- Prisma generation errors
- BullMQ worker errors
- CORS mistakes
- Next.js build errors
- NestJS startup errors
- Missing dependencies
- Incorrect workspace scripts
- Open Jest handles
- Broken health endpoints
- CI-only failures

Running locally is safe when only local development credentials and local Docker services are used.

```text

```

Requirements:

- Use `#!/usr/bin/env bash`
- Use `set -euo pipefail`
- Run from repository root regardless of the caller’s current directory
- Never echo secrets
- Check required tools
- Check required files

- Validate Docker Compose configuration

- Run lint
- Run type-check
- Run unit tests
- Run API E2E tests
- Run web component tests
- Run production builds
- Print a clear result
- Never remove volumes
- Never reset the database

Use:

```bash

```

If the lockfile is not synchronized, fail and report it rather than silently rewriting it.

### Required Tool Checks


### Required Infrastructure Checks

```bash

  -f infrastructure/docker/compose.dev.yml \
```

```bash

  -f infrastructure/docker/compose.dev.yml \
  exec -T redis redis-cli ping
```

```bash

  http://127.0.0.1:9000/minio/health/live
```

Mailpit:

```bash

  http://127.0.0.1:8025/
```

Run the existing equivalent commands:

## 12. Smoke-Test Script

Create:

```text
infrastructure/scripts/smoke-test.sh
```

Requirements:

- Use `set -euo pipefail`
- Do not start production processes
- Start the API and web development servers as background processes
- Capture process IDs
- Capture logs in a temporary directory
- Wait for both applications
- Call required routes
- Stop both processes on exit
- Use a cleanup trap
- Print useful failure logs
- Do not leave background processes running
- Do not expose secrets

### API Checks

Verify:

```text
GET http://127.0.0.1:3001/api/v1
GET http://127.0.0.1:3001/api/v1/health
GET http://127.0.0.1:3001/api/v1/health/ready
GET http://127.0.0.1:3001/api/docs
```

Verify unknown API route returns 404.

If the system queue endpoint exists and is enabled:

```text
POST http://127.0.0.1:3001/api/v1/system/queue/ping
```

The script must not fail when the endpoint is intentionally disabled. It must report whether the check was skipped.

### Web Checks

Verify:

```text
GET http://127.0.0.1:3000/
GET http://127.0.0.1:3000/status
```

Verify unknown web route displays the custom not-found behavior.

### Process Cleanup

Use a trap similar to:

```bash
cleanup() {
  kill "$API_PID" "$WEB_PID" 2>/dev/null || true
}
trap cleanup EXIT
```

Wait for services with bounded retries. Do not wait forever.

## 20. Problem Handling

### Port Conflict

If a port is already in use:

1. Identify the process.
2. Do not kill unrelated user processes automatically.
3. Report the conflict.
4. Use configured alternate local ports only if the project supports them safely.
5. Update `.env`, not committed examples, when using a temporary local override.

### Docker Not Running

- Mark runtime verification `BLOCKED`.
- Do not claim services passed.
- Static CI configuration may still be completed.

### Lockfile Mismatch

- Do not bypass `--frozen-lockfile`.
- Run normal `pnpm install` only if dependency changes in this task legitimately require a lockfile update.
- Then rerun frozen installation.

### Failing Existing Tests

- Determine whether the failure is caused by this task or an earlier incomplete task.
- Fix small foundation defects only when directly required for CI verification.
- Do not expand into business development.
- Document every compatibility fix.

### Playwright Environment Failure

- Report browser or system dependency issue.
- Do not claim E2E success.
- Upload/report logs where possible.
