## 6. Business Rules

1. Only authenticated active candidate accounts can access this feature.
2. A candidate can view only their own account-security summary.
3. Current password is required for password change.
4. Current password verification uses the existing Argon2id configuration.
5. Wrong current password returns a generic controlled error.
6. New password must follow the shared password policy.
7. New password must not equal the current password.
8. New password and confirmation must match.
9. Password update must be atomic.
10. Successful password change revokes all other active sessions.
11. The current session remains active.
12. The current refresh token may remain valid or be rotated according to the existing secure architecture.
13. Other revoked sessions cannot refresh or access protected APIs.
14. Passwords and hashes must never appear in logs, audit metadata, or API responses.
15. Security summary must not expose token hashes, session IDs, IP addresses, raw user agents, or internal role IDs.
16. Account email may be shown as read-only.
17. Account status and email verification status may be shown safely.
18. Active-session count may be shown.
19. Last password-change time may be shown only when stored explicitly; never infer it from generic `updatedAt`.
20. Suspended or deleted accounts cannot change passwords.
21. Rate limiting applies to password-change attempts.
22. API authorization is authoritative.
23. All timestamps use UTC.
24. Unexpected errors must not expose Prisma or stack-trace details.
25. A blocked task must not create a Git commit.
26. A Git commit occurs only after all mandatory checks pass.

## 7. Account Security Summary

```ts
interface CandidateAccountSecuritySummary {
  email: string;
  accountStatus: 'ACTIVE';
  emailVerified: boolean;
  activeSessionCount: number;
  currentSessionCreatedAt: string;
  currentSessionLastUsedAt: string | null;
  passwordLastChangedAt: string | null;
  securityLinks: {
    sessions: '/settings/security/sessions';
    privacy: '/settings/privacy';
  };
}
```

Do not expose:

- password hash
- refresh-token hash
- token-family ID
- raw session IDs
- IP address
- raw user agent
- role/permission IDs
- audit-log details
- reset-token data

If `passwordLastChangedAt` does not exist, either add the smallest safe field such as `passwordChangedAt` to `User`, or omit the field and document the choice.

## 8. Change Password Contract

Suggested request:

```ts
{
  currentPassword: string;
  newPassword: string;
  confirmNewPassword: string;
}
```

Suggested response:

```ts
{
  changed: true;
  revokedOtherSessionCount: number;
}
```

Validation:

- `currentPassword`: required, bounded length, no trimming.
- `newPassword`: required, shared password policy, no trimming, different from current password.
- `confirmNewPassword`: required and exact match.

Never return access tokens, refresh tokens, hashes, or session IDs.

## 10. API Contracts

### 10.1 Get Account Security Summary

```http
GET /api/v1/candidates/me/account-security
```

Authenticated candidate only.

Success: `200 OK`

Controlled errors:

```text
401 AUTH_ACCESS_TOKEN_INVALID
403 CANDIDATE_ROLE_REQUIRED
403 AUTH_ACCOUNT_UNAVAILABLE
500 INTERNAL_SERVER_ERROR
```

### 10.2 Change Password

```http
POST /api/v1/auth/change-password
```

Authenticated active account only.

Request:

```json
{
  "currentPassword": "CurrentPass#2026",
  "newPassword": "NewStrongPass#2026",
  "confirmNewPassword": "NewStrongPass#2026"
}
```

Success: `200 OK`

```json
{
  "changed": true,
  "revokedOtherSessionCount": 2
}
```

Controlled errors:

```text
400 AUTH_CHANGE_PASSWORD_VALIDATION_FAILED
400 AUTH_NEW_PASSWORD_REUSED
401 AUTH_CURRENT_PASSWORD_INVALID
401 AUTH_ACCESS_TOKEN_INVALID
403 AUTH_ACCOUNT_UNAVAILABLE
429 AUTH_CHANGE_PASSWORD_RATE_LIMITED
500 INTERNAL_SERVER_ERROR
```

Swagger must document authentication, password policy, current-session preservation, other-session revocation, and controlled errors.

## 11. Password Change Service

The service must:

1. Read authenticated user ID and current session ID.
2. Confirm active account.
3. Load password hash securely.
4. Verify current password.
5. Validate the new password.
6. Confirm the new password differs from the current password.
7. Hash the new password with existing Argon2id settings.
8. Begin a transaction.
9. Update password hash.
10. Set `passwordChangedAt` when supported.
11. Revoke all other active sessions.
12. Preserve the current session.
13. Rotate current refresh token only if required by existing architecture.
14. Commit.
15. Record audit event.
16. Return a safe result.

If any step fails, do not partially update the password or sessions.

## 12. Rate Limiting

Reuse existing infrastructure.

Recommended:

```text
5 change-password attempts per authenticated user per 15 minutes
```

Add IP-based protection only when the existing architecture supports it. Do not expose counters or password-validity details.
