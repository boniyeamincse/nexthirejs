```text

        ↓
```

## 5. Business Rules

Implement these exact rules:

1. Only verified `ACTIVE` accounts can log in successfully.
2. Email is trimmed and normalized before lookup.
3. Invalid email/password returns one generic credential error.
4. Do not reveal whether the email exists.
5. `PENDING_VERIFICATION` accounts receive a controlled unverified-account error.
6. `SUSPENDED` accounts receive a controlled unavailable-account error.
7. `DELETED` or equivalent non-login accounts must not authenticate.
8. Password verification uses the existing Argon2id hash.
9. Access tokens are short-lived.
10. Refresh tokens are longer-lived and rotated on every successful refresh.
11. Store only the refresh-token hash.
12. Raw refresh tokens must never be stored or logged.
13. Refresh tokens must be delivered to web clients through a secure HttpOnly cookie.
14. Frontend JavaScript must not read or persist the refresh token.
15. Access-token storage must follow the chosen documented web architecture.
16. A refresh token is bound to one session.
17. Refresh-token replay or reuse revokes the affected session or token family.
18. Current-device logout revokes the current session.
19. A revoked, expired, or replayed refresh token cannot create a new access token.
20. Authentication errors must not expose token hashes, password hashes, database details, or stack traces.
21. Every successful and failed sensitive auth action must use the audit foundation according to its failure-mode policy.
22. Cookies must be environment-aware and never hardcode production domains.
23. CORS must use explicit origins and credentials when cookie auth is enabled.
24. Existing public routes must remain public.
25. Protected routes must be enforced by the API guard, not only hidden in the frontend.
26. Timestamps use UTC.
27. Login success must not automatically complete candidate profile onboarding.
28. Authentication responses must not expose internal role IDs or permission IDs.

## 6. Token and Session Design

Inspect existing token/security conventions before implementation.

### 6.1 Access token

Recommended properties:

```text
type: signed JWT
lifetime: 10–15 minutes
audience: NextHire API
issuer: NextHire
subject: user ID
session ID claim: required
roles: role codes only
permissions: omit until permission foundation is implemented, or include only when sourced authoritatively
```

Keep the payload minimal.

Suggested logical claims:

```ts
{
  sub: string;
  sid: string;
  roles: string[];
  type: 'access';
  iat: number;
  exp: number;
  iss: string;
  aud: string;
}
```

Do not include:

- password hash
- refresh-token hash
- email-verification token
- private profile data
- internal database role IDs
- full permissions object
- secret configuration values

### 6.2 Refresh token

Use a cryptographically secure random token or an approved signed token design.

Preferred logical design:

```text
raw token: at least 32 random bytes
transport: base64url
stored value: SHA-256 or stronger hash
bound fields: session ID + token family/rotation identifier
```

Requirements:

- rotate on every successful refresh;
- invalidate the previous token atomically;
- detect replay/reuse;
- never log raw token;
- never return raw refresh token in JSON for the browser flow.

### 6.3 Access-token storage

Choose one repository-consistent approach and document it:

#### Preferred approach

- access token kept in memory by the web auth client;
- refresh token stored in secure HttpOnly cookie;
- page reload uses refresh endpoint to obtain a new access token;
- protected API calls use `Authorization: Bearer <access token>`.

Do not store access tokens in local storage.

If the repository already uses a backend-for-frontend secure cookie design, follow it instead of creating a competing pattern.

### 6.4 Cookie requirements

Refresh cookie must use:

```text
HttpOnly: true
Secure: true in production
SameSite: Lax by default
Path: refresh endpoint scope or approved auth path
Max-Age/Expires: aligned with refresh expiry
Domain: omitted unless explicitly configured
```

When web and API run cross-site in a real environment, reassess `SameSite=None; Secure` and CSRF requirements. Document the deployment assumption.

Suggested environment variables:

```text
AUTH_ACCESS_TOKEN_TTL_MINUTES=15
AUTH_REFRESH_TOKEN_TTL_DAYS=30
AUTH_JWT_ISSUER=nexthire
AUTH_JWT_AUDIENCE=nexthire-api
AUTH_ACCESS_TOKEN_SECRET=<local-only-secret>
AUTH_REFRESH_COOKIE_NAME=nexthire_refresh
WEB_APP_URL=http://localhost:3000
```

Use existing names when already defined. Add safe placeholders only to `.env.example`.

## 12. Refresh Rotation and Replay Detection

Implement rotation atomically.

Logical successful refresh:

```text
read raw refresh cookie
  ↓
hash token
  ↓
find active session/token record
  ↓
validate expiry and status
  ↓
generate new refresh token
  ↓
replace stored hash atomically
  ↓
update lastUsedAt
  ↓
issue new access token
  ↓
set replacement refresh cookie
```

Replay/reuse behavior:

- a previously rotated token must not succeed;
- mark the affected session `COMPROMISED` or `REVOKED`;
- clear the cookie;
- record `auth.refresh_token.reused`;
- do not expose internal detection details.

If robust replay detection requires a token-history table, add the smallest secure versioning model necessary. Do not weaken rotation merely to avoid a migration.
