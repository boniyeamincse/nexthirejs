## 6. Business Rules

Implement these exact rules:

1. Only authenticated users with the `candidate` role can use this feature.
2. A candidate can access only their own skills and languages.
3. A candidate may have multiple skills.
4. A candidate may have multiple languages.
5. Maximum 50 skills per candidate.
6. Maximum 20 languages per candidate.
7. Skill names are unique per candidate, case-insensitively.
8. Language names are unique per candidate, case-insensitively.
9. Skill name is required.
10. Skill level is required.
11. Language name is required.
12. Speaking, reading, and writing proficiency values are required.
13. Arrays or list inputs must reject duplicate IDs.
14. Ordering is candidate-controlled.
15. Reorder input must contain only owned record IDs.
16. Empty optional strings normalize to `null`.
17. Audit metadata must not contain skill names or language names.
18. API authorization is authoritative.
19. Suspended or deleted accounts cannot modify records.
20. All timestamps use UTC.
21. Cross-user access returns controlled not-found/forbidden behavior.
22. Request bodies must not accept arbitrary user IDs.
23. Profile completion is recalculated after create/update/delete.
24. The backend completion result is authoritative.
25. A blocked task must not create a commit.
26. A Git commit occurs only after all mandatory checks pass.

## 9. Database Design

Inspect existing Prisma conventions.

Implement the logical equivalent of:

```prisma
enum SkillLevel {
  BEGINNER
  DEVELOPING
  INTERMEDIATE
  ADVANCED
  EXPERT
}

enum LanguageProficiency {
  BASIC
  CONVERSATIONAL
  PROFESSIONAL
  FLUENT
  NATIVE
}

model CandidateSkill {
  id                String     @id @default(uuid()) @db.Uuid
  userId            String     @db.Uuid
  name              String     @db.VarChar(100)
  normalizedName    String     @db.VarChar(100)
  level             SkillLevel
  yearsOfExperience Decimal?   @db.Decimal(4,1)
  sortOrder         Int        @default(0)
  createdAt         DateTime   @default(now())
  updatedAt         DateTime   @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, normalizedName])
  @@index([userId, sortOrder])
}

model CandidateLanguage {
  id             String              @id @default(uuid()) @db.Uuid
  userId         String              @db.Uuid
  name           String              @db.VarChar(100)
  normalizedName String              @db.VarChar(100)
  speaking       LanguageProficiency
  reading        LanguageProficiency
  writing        LanguageProficiency
  sortOrder      Int                 @default(0)
  createdAt      DateTime            @default(now())
  updatedAt      DateTime            @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, normalizedName])
  @@index([userId, sortOrder])
}
```

If the repository avoids Prisma `Decimal`, use a validated integer representation such as months of experience. Document the decision.

### Ordering

- assign next sort order on create;
- list ascending by sort order, then name;
- reorder atomically;
- normalize sort order after deletion when practical.

### Migration safety

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not run `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
- Document migration results.

```http
GET    /api/v1/candidates/me/languages

DELETE /api/v1/candidates/me/languages/:languageId

```

### Success statuses

- list: `200 OK`
- create: `201 Created`
- update: `200 OK`
- delete: `204 No Content`

