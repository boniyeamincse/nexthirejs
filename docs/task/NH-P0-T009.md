## 2. Objective

Create a centralized, typed, fail-fast environment validation and secrets-handling foundation for the NextHire monorepo.

The task must ensure that:

- The NestJS API validates all required environment variables before startup.
- The Next.js web app validates public environment variables during development and build.
- Docker development variables remain documented and compatible.
- Secrets and public configuration are clearly separated.
- No real credentials are committed.
- Test environments can use safe isolated values.
- Configuration errors produce clear messages without exposing secret values.

## 8. API Environment Schema

Use Zod from the shared validation package when available. If importing the shared package creates a circular build problem, use `zod` directly in the API and document the reason.

Create a schema covering all API/runtime variables currently used by the repository.

At minimum validate:

```text
NODE_ENV
API_HOST
API_PORT
API_GLOBAL_PREFIX
API_DEFAULT_VERSION
API_CORS_ORIGINS
API_DOCS_ENABLED
API_BODY_LIMIT
DATABASE_URL
POSTGRES_HOST
POSTGRES_PORT
POSTGRES_DB
POSTGRES_USER
POSTGRES_PASSWORD
REDIS_HOST
REDIS_PORT
REDIS_USERNAME
REDIS_PASSWORD
REDIS_DB
REDIS_TLS
REDIS_CONNECT_TIMEOUT_MS
REDIS_COMMAND_TIMEOUT_MS
REDIS_MAX_RETRIES_PER_REQUEST
BULLMQ_PREFIX
BULLMQ_DEFAULT_ATTEMPTS
BULLMQ_DEFAULT_BACKOFF_MS
BULLMQ_REMOVE_ON_COMPLETE
BULLMQ_REMOVE_ON_FAIL
SYSTEM_QUEUE_TEST_ENABLED
MINIO_ENDPOINT
MINIO_ROOT_USER
MINIO_ROOT_PASSWORD
MINIO_DEFAULT_BUCKET
MAIL_HOST
MAIL_PORT
MAIL_FROM_ADDRESS
MAIL_FROM_NAME
```

Validation requirements:

- `NODE_ENV`: enum of `development`, `test`, `staging`, `production`
- Ports: integer from 1 to 65535
- Database/Redis DB numbers: non-negative integer
- Timeouts: positive integer
- Retry counts: non-negative integer
- Booleans: parse explicit string booleans safely
- URLs: valid URLs
- Email addresses: valid email format
- Global prefix: safe path segment without leading/trailing slash after normalization
- API version: positive integer represented as a normalized string or number according to current bootstrap code
- CORS origins: comma-separated valid origins; wildcard must be rejected when credentials are enabled
- Body limit: validate a safe format such as `1mb`, `500kb`, or another explicitly supported unit
- Password-like fields: required in development where the corresponding service is used, but values must never be shown in error output

Do not silently coerce arbitrary invalid strings.

## 9. API Validation Function

Create a validation function compatible with NestJS `ConfigModule.forRoot({ validate })`.

Required behavior:

1. Parse `process.env` using the Zod schema.
2. Return normalized typed values.
3. Throw one clear startup error when validation fails.
4. Include invalid variable names and reasons.
5. Never include secret values.
6. Group all detected validation issues rather than failing on only the first issue.
7. Work in unit tests without mutating global state unexpectedly.

Secret variable names must include at minimum:

```text
POSTGRES_PASSWORD
REDIS_PASSWORD
MINIO_ROOT_PASSWORD
DATABASE_URL
```

Future secret names should be easy to add.

Validate only browser-safe variables:

```text
NEXT_PUBLIC_APP_NAME
NEXT_PUBLIC_APP_URL
NEXT_PUBLIC_API_BASE_URL
```

Requirements:

- Use Zod
- Valid non-empty app name
- Valid URLs
- Export a typed immutable object
- No server secret import
- No access to database, Redis, MinIO root credentials, or mail credentials
- Validation must run during development/build where the module is imported
- Error must name the invalid public variable
- Error must not print unrelated environment values

Do not make all public variables optional in production.

A safe rule is:

- Development/test: documented defaults may be allowed
- Staging/production: required explicit values

Implement this without introducing browser access to `NODE_ENV` beyond standard Next.js behavior.
