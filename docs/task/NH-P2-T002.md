## 7. Question Types

Implement these question types:

```text
SINGLE_CHOICE
MULTIPLE_CHOICE
TRUE_FALSE
SHORT_TEXT
```

Do not implement essay, file upload, coding, matching, ordering, or numeric-expression questions in this task.

### Type Rules

#### `SINGLE_CHOICE`

- minimum 2 options
- maximum 10 options
- exactly 1 correct option

#### `MULTIPLE_CHOICE`

- minimum 2 options
- maximum 10 options
- at least 2 correct options
- not all options may be correct

#### `TRUE_FALSE`

- exactly 2 options
- labels normalized to `True` and `False`
- exactly 1 correct option
- prevent arbitrary extra options

#### `SHORT_TEXT`

- no selectable options
- one or more accepted answers
- maximum 10 accepted answers
- accepted answers are normalized for matching metadata only
- actual candidate matching/scoring is deferred

## 8. Question Status and Lifecycle

Implement:

```text
DRAFT
ACTIVE
ARCHIVED
```

Supported transitions:

```text
DRAFT -> ACTIVE
ACTIVE -> ARCHIVED
ARCHIVED -> ACTIVE
DRAFT -> ARCHIVED
```

Rules:

1. New questions default to `DRAFT`.
2. Only `ACTIVE` questions may be assigned to assessments in future tasks.
3. Archived questions remain stored.
4. Questions referenced by future assessment mappings must not be hard deleted.
5. Hard deletion is out of scope; use archive behavior.
6. Restoring an archived question returns it to `ACTIVE` unless repository policy requires `DRAFT`.
7. Lifecycle helper must be shared and tested.
8. Invalid transitions return controlled errors.

## 9. Database Design

Inspect existing Prisma conventions and adapt this logical design.

```prisma
enum AssessmentQuestionType {
  SINGLE_CHOICE
  MULTIPLE_CHOICE
  TRUE_FALSE
  SHORT_TEXT
}

enum AssessmentQuestionStatus {
  DRAFT
  ACTIVE
  ARCHIVED
}

model AssessmentQuestion {
  id                  String                   @id @default(uuid()) @db.Uuid
  categoryId          String                   @db.Uuid
  type                AssessmentQuestionType
  status              AssessmentQuestionStatus @default(DRAFT)
  difficulty          AssessmentDifficulty
  prompt              String                   @db.Text
  explanation         String?                  @db.Text
  acceptedAnswersJson Json?
  tags                String[]
  sourceReference     String?                  @db.VarChar(300)
  estimatedSeconds    Int?
  createdById         String?                  @db.Uuid
  updatedById         String?                  @db.Uuid
  archivedAt          DateTime?
  createdAt           DateTime                 @default(now())
  updatedAt           DateTime                 @updatedAt

  category AssessmentCategory       @relation(fields: [categoryId], references: [id], onDelete: Restrict)
  options  AssessmentQuestionOption[]

  @@index([categoryId, status])
  @@index([type, difficulty, status])
  @@index([createdAt])
}

model AssessmentQuestionOption {
  id         String   @id @default(uuid()) @db.Uuid
  questionId String   @db.Uuid
  label      String   @db.Text
  isCorrect  Boolean  @default(false)
  sortOrder  Int
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  question AssessmentQuestion @relation(fields: [questionId], references: [id], onDelete: Cascade)

  @@unique([questionId, sortOrder])
  @@index([questionId])
}
```

If PostgreSQL string arrays are not used elsewhere, replace `tags` with a normalized tag model or JSON according to repository conventions. Document the choice.

### Field Rules

#### prompt

- required
- trimmed
- 5–5,000 characters
- plain text or existing sanitized rich-text format only
- no unsafe HTML

#### explanation

- optional
- maximum 5,000 characters
- not exposed to candidates until future result workflows explicitly allow it

#### tags

- optional
- maximum 20 tags
- each 2–50 characters
- trimmed
- case-normalized
- duplicates removed

#### sourceReference

- optional
- maximum 300 characters
- internal management metadata

#### estimatedSeconds

- optional
- minimum 10
- maximum 3,600

### Migration Safety

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not use `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
- Validate clean migration order.
- Document migration results.

## 10. Category Management Rules

Category fields:

```ts
{
  name: string;
  slug: string;
  description?: string | null;
  sortOrder?: number;
  isActive?: boolean;
}
```

Rules:

1. Name required, trimmed, 2–120 characters.
2. Slug required, lowercase normalized, globally unique.
3. Description optional, maximum 1,000 characters.
4. Inactive categories cannot receive new active questions.
5. Deactivating a category does not archive existing questions.
6. Category with assessments or questions cannot be hard deleted.
7. Reorder input must contain only existing category IDs.
8. Reorder is atomic.
9. Candidate catalog continues to hide inactive categories.
10. Seeded categories remain editable according to permissions.

Use deactivate/reactivate instead of hard delete.
