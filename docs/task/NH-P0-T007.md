## 8. Shared TypeScript Configuration

### 8.1 `@nexthire/tsconfig`

Create:

```text
packages/tsconfig/base.json
packages/tsconfig/nestjs.json
packages/tsconfig/nextjs.json
```

### Base configuration requirements

Use strict, modern, reusable settings:

- `strict: true`
- `noUncheckedIndexedAccess: true`
- `noImplicitOverride: true`
- `noFallthroughCasesInSwitch: true`
- `forceConsistentCasingInFileNames: true`
- `skipLibCheck: true`
- `esModuleInterop: true`
- `resolveJsonModule: true`
- `isolatedModules: true` only where compatible
- Modern ECMAScript target supported by current Node.js LTS
- No weakening of existing application type safety

Do not force application-specific path aliases from the shared base config.

### NestJS configuration requirements

Extend `base.json` and include settings compatible with NestJS decorators and compilation.

Do not break the existing API build.

### Next.js configuration requirements

Extend `base.json` and remain compatible with Next.js-generated TypeScript settings.

Do not remove Next.js-required plugin configuration.

### Application integration

Update:

```text
apps/api/tsconfig.json
apps/web/tsconfig.json
```

so they extend the appropriate shared config while preserving application-specific settings.

Do not replace working framework-required options blindly.

## 9. Shared Types Package

Create `@nexthire/types` as a type-only package where possible.

### 9.1 API Types

Create reusable types equivalent to:

```typescript
export interface ApiSuccessResponse<TData, TMeta = undefined> {
  success: true;
  data: TData;
  meta?: TMeta;
}

export interface ApiErrorDetail {
  field?: string;
  code: string;
  message: string;
}

export interface ApiErrorResponse {
  success: false;
  message: string;
  errors?: ApiErrorDetail[];
  requestId?: string;
}

export type ApiResponse<TData, TMeta = undefined> =
  ApiSuccessResponse<TData, TMeta> | ApiErrorResponse;
```

Rules:

- No `any`
- Generic defaults must be safe
- Do not create framework-specific response objects
- Do not include secrets or infrastructure internals

### 9.2 Pagination Types

Create:

```typescript
export interface CursorPaginationMeta {
  nextCursor: string | null;
  hasMore: boolean;
  perPage: number;
}

export interface CursorPaginatedData<TItem> {
  items: TItem[];
  pagination: CursorPaginationMeta;
}
```

### 9.3 Utility Types

Create minimal utilities:

```typescript
export type Nullable<T> = T | null;
export type Optional<T> = T | undefined;
export type ValueOf<TObject> = TObject[keyof TObject];
```

Do not add speculative utility types.

### 9.4 Exports

Export everything from:

```text
src/index.ts
```

The package must support TypeScript consumers without importing internal file paths.

## 10. Shared Constants Package

Create `@nexthire/constants` with immutable foundational constants.

### 10.1 Supported Countries

Use only current initial target markets:

```typescript
export const SUPPORTED_COUNTRIES = ['BD', 'PK', 'IN'] as const;
```

Provide a typed map containing:

- ISO code
- English name
- Calling code
- Default currency
- Default timezone identifier

Use values:

- Bangladesh: `BD`, `+880`, `BDT`, `Asia/Dhaka`
- Pakistan: `PK`, `+92`, `PKR`, `Asia/Karachi`
- India: `IN`, `+91`, `INR`, `Asia/Kolkata`

### 10.2 Supported Currencies

```typescript
export const SUPPORTED_CURRENCIES = ['BDT', 'PKR', 'INR', 'USD'] as const;
```

Provide display metadata without performing currency conversion.

### 10.3 Supported Languages

```typescript
export const SUPPORTED_LANGUAGES = ['en', 'bn', 'ur', 'hi'] as const;
```

Metadata must include:

- Code
- English name
- Native name
- Text direction

Urdu must use `rtl`; all others use `ltr`.

### 10.4 HTTP Constants

Create safe generic constants:

```typescript
export const DEFAULT_PAGE_SIZE = 20;
export const MAX_PAGE_SIZE = 50;
export const REQUEST_ID_HEADER = 'x-request-id';
export const IDEMPOTENCY_KEY_HEADER = 'idempotency-key';
```

Do not duplicate framework HTTP status code enums.

### 10.5 Immutability

Use `as const` and readonly data where appropriate.

Do not expose mutable arrays or records unintentionally.

## 11. Shared Validation Package

Create `@nexthire/validation` using Zod.

Install:

```bash
pnpm --filter @nexthire/validation add zod
```

### 11.1 Common Schemas

Create reusable schemas:

- `uuidSchema`
- `emailSchema`
- `nonEmptyTrimmedStringSchema`
- `countryCodeSchema`
- `currencyCodeSchema`
- `languageCodeSchema`

Requirements:

- Email values are trimmed and lowercased
- Non-empty strings trim whitespace
- Country, currency, and language schemas derive from shared constants
- Do not duplicate literal lists manually where avoidable
- Validation errors must be clear

### 11.2 Cursor Pagination Schema

Create:

```typescript
cursorPaginationQuerySchema;
```

Fields:

- `cursor`: optional non-empty string
- `perPage`: optional coerced integer

Rules:

- Default `perPage` is `DEFAULT_PAGE_SIZE`
- Minimum `1`
- Maximum `MAX_PAGE_SIZE`
- Reject decimals
- Reject non-numeric invalid values
- Do not allow zero or negative values

Export inferred TypeScript types from the schemas where useful.

### 11.3 No Framework Coupling

The package must not import:

- NestJS
- Next.js
- Prisma
- React
- Node-specific APIs

It must remain portable TypeScript.

## 12. Package Build Strategy

Use a simple build tool compatible with the workspace.

Preferred:

```text
tsup
```

Install it as a development dependency in packages that emit JavaScript:

```bash
pnpm --filter @nexthire/types add -D tsup typescript
pnpm --filter @nexthire/constants add -D tsup typescript vitest
pnpm --filter @nexthire/validation add -D tsup typescript vitest
```

Requirements:

- ESM output
- Type declarations
- Source maps
- Clean output before build
- `dist/` ignored by Git
- Explicit package exports
- No CommonJS unless the existing repository requires dual output

Recommended package scripts:

```json
{
  "build": "tsup src/index.ts --format esm --dts --sourcemap --clean",
  "dev": "tsup src/index.ts --format esm --dts --sourcemap --watch",
  "typecheck": "tsc --noEmit",
  "test": "vitest run",
  "test:watch": "vitest",
  "clean": "rimraf dist coverage"
}
```

`@nexthire/tsconfig` does not need a build output.

If `rimraf` is already available through the workspace, reuse it appropriately. Otherwise install it only where required.
