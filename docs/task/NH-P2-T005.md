## 3. Objective

Implement this end-to-end flow:

```text
candidate completes draft answers
        ↓
candidate reviews answered/unanswered count
        ↓
candidate confirms final submission
        ↓
backend locks the attempt
        ↓
backend scores immutable snapshot answers
        ↓
score, percentage, pass/fail, and question statistics are persisted
        ↓
attempt status becomes SUBMITTED
        ↓
further answer edits are blocked
        ↓
candidate receives a safe result summary
        ↓
all checks pass
        ↓
one Git commit is created
```

Also handle overdue attempts safely:

```text
deadline passes
        ↓
backend finalizes according to documented expiry policy
        ↓
attempt is no longer editable
        ↓
scoring is deterministic and idempotent
```

## 9. Scoring Rules

Create one scoring strategy per supported type.

### Single Choice

Correct when:

```text
selected option set equals the one correct option
```

Award:

```text
full points or zero
```

### Multiple Choice

Correct when:

```text
selected option set exactly equals the full correct option set
```

Award:

```text
full points or zero
```

No partial credit.

### True/False

Correct when:

```text
selected option is the correct snapshot option
```

Award:

```text
full points or zero
```

### Short Text

Normalize candidate answer and accepted snapshot answers using one documented function.

Recommended normalization:

1. Unicode normalization
2. trim leading/trailing whitespace
3. collapse repeated internal whitespace
4. case-fold/lowercase
5. preserve punctuation by default
6. exact normalized equality

Award:

```text
full points when one accepted answer matches, otherwise zero
```

Do not implement semantic or fuzzy matching.

### Unanswered

Award zero and classify as unanswered.

## 11. Deadline Finalization Policy

Document and implement one clear policy.

Recommended policy:

1. Explicit submission before or at the deadline uses `CANDIDATE_SUBMITTED`.
2. After deadline, answer writes are blocked.
3. When an overdue `IN_PROGRESS` attempt is read or submission is attempted:
   - finalize the current saved answers;
   - set status to `SUBMITTED`;
   - set reason to `DEADLINE_REACHED`;
   - persist automatic score.
4. Do not leave overdue attempts indefinitely unscored.
5. Finalization must be idempotent and race-safe.
6. Client countdown cannot decide final score.
7. A future worker may proactively finalize overdue attempts, but lazy finalization is mandatory here.

If repository policy requires keeping `EXPIRED`, then:

- finalize score;
- retain `EXPIRED`;
- persist result fields;
- document the deviation.

Choose one approach and apply it consistently across API, UI, tests, and documentation.

## 15. Score Calculation

Recommended formulas:

```text
scoreEarned = sum(awardedPoints)
scorePossible = attempt.totalPointsSnapshot
percentage =
  scorePossible > 0
    ? round((scoreEarned / scorePossible) * 100, 2)
    : 0
```

Pass:

```text
percentage >= passingScoreSnapshot
```

Rules:

- use decimal-safe arithmetic;
- avoid JavaScript floating-point accumulation when repository uses Decimal;
- never trust client totals;
- verify assignment points sum to snapshot total;
- result counts must reconcile with question count.
