## 5. Business Rules

Implement these exact rules:

1. The endpoint registers candidates only.
2. Email is trimmed and normalized to lowercase before lookup and storage.
3. Email uniqueness is enforced by the database.
4. Password is never stored or logged in plain text.
5. Password is hashed using Argon2id.
6. `confirmPassword` is validated but never stored.
7. Terms acceptance must be exactly `true`.
8. New accounts start as `PENDING_VERIFICATION`.
9. Every successful account receives the system role `candidate`.
10. User creation and role assignment occur in one Prisma transaction.
11. Duplicate normalized email returns a controlled conflict.
12. A successful response contains no password, hash, token, cookie, role ID, or private database metadata.
13. Registration does not log the user in.
14. Registration writes an audit event through the existing audit foundation.
15. Audit metadata must not contain the email address or password data.
16. Frontend validation improves usability; backend validation remains authoritative.
17. The frontend must not claim that a verification email has already been sent.
18. Repeated form submission is prevented while a request is pending.
19. Unexpected errors must not expose Prisma, stack-trace, or infrastructure details.
20. All timestamps are returned as ISO 8601 UTC strings.

- at least one uppercase English letter
- at least one lowercase English letter
- at least one number
- at least one special character
- must not equal the normalized email
- must match `confirmPassword`

## 7. Database Design

Inspect existing Prisma naming, UUID, mapping, timestamp, and schema conventions before implementation.

Implement the logical equivalent of:

```prisma
enum UserStatus {
  PENDING_VERIFICATION
  ACTIVE
  SUSPENDED
  DELETED
}

model User {
  id              String      @id @default(uuid()) @db.Uuid
  email           String      @unique @db.VarChar(320)
  passwordHash    String
  status          UserStatus  @default(PENDING_VERIFICATION)
  emailVerifiedAt DateTime?
  createdAt       DateTime    @default(now())
  updatedAt       DateTime    @updatedAt

  roles           UserRole[]
}

model Role {
  id          String     @id @default(uuid()) @db.Uuid
  code        String     @unique @db.VarChar(50)
  name        String     @db.VarChar(100)
  description String?
  isSystem    Boolean    @default(true)
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt

  users       UserRole[]
}

model UserRole {
  userId     String   @db.Uuid
  roleId     String   @db.Uuid
  assignedAt DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  role Role @relation(fields: [roleId], references: [id], onDelete: Restrict)

  @@id([userId, roleId])
  @@index([roleId])
}
```

Adapt physical names only when existing conventions require it.

## 10. Backend Implementation

Use the existing NestJS feature structure.

The registration service must:

1. Normalize the email.
2. Validate final input.
3. Check for an existing account for an early controlled response.
4. Hash the password with Argon2id.
5. Resolve the `candidate` role.
6. Start a Prisma transaction.
7. Create the pending-verification user.
8. Assign the candidate role.
9. Commit the transaction.
10. Record `auth.candidate.registered` through the audit service.
11. Return only safe response fields.
12. Catch the database unique-constraint race and return `409`.
13. Avoid logging raw request bodies, passwords, or hashes.

### Password hashing

Use an approved Argon2 package and explicitly select Argon2id when required.

Requirements:

- use secure library defaults unless the repository already documents approved parameters;
- keep hashing testable;
- verify hashes in tests rather than comparing fixed hash strings;
- do not log the generated hash.

### Audit event

Record:

```text
action: auth.candidate.registered
actorType: USER
actorUserId: newly created user ID
targetType: user
targetId: newly created user ID
outcome: SUCCESS
metadata:
  registrationChannel: web
  accountStatus: PENDING_VERIFICATION
```

Do not include email, password, hash, token, user agent, or full request body in metadata.

Use required audit mode if the existing audit design defines this identity event as mandatory. Document the chosen mode.
