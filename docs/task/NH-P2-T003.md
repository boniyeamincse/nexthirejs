## 8. Assessment Sections

Suggested model:

```prisma
model AssessmentSection {
  id           String   @id @default(uuid()) @db.Uuid
  assessmentId String   @db.Uuid
  title        String   @db.VarChar(200)
  description  String?  @db.Text
  instructions String?  @db.Text
  sortOrder    Int
  isRequired   Boolean  @default(true)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  assessment Assessment                    @relation(fields: [assessmentId], references: [id], onDelete: Cascade)
  questions  AssessmentQuestionAssignment[]

  @@unique([assessmentId, sortOrder])
  @@index([assessmentId])
}
```

Rules:

1. Draft may have zero sections.
2. Published assessment requires at least one section.
3. Title 2–200 characters.
4. Description/instructions maximum 5,000.
5. Maximum 20 sections.
6. Ordering is deterministic and transactional.
7. Reorder input contains only owned section IDs.
8. Deleting a section removes assignments, not question-bank records.
9. Deleting a non-empty section requires UI confirmation.
10. Retired assessment is read-only.

## 9. Question Assignment

Suggested model:

```prisma
model AssessmentQuestionAssignment {
  id           String   @id @default(uuid()) @db.Uuid
  assessmentId String   @db.Uuid
  sectionId    String   @db.Uuid
  questionId   String   @db.Uuid
  sortOrder    Int
  points       Decimal  @db.Decimal(8,2)
  isRequired   Boolean  @default(true)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  assessment Assessment         @relation(fields: [assessmentId], references: [id], onDelete: Cascade)
  section    AssessmentSection  @relation(fields: [sectionId], references: [id], onDelete: Cascade)
  question   AssessmentQuestion @relation(fields: [questionId], references: [id], onDelete: Restrict)

  @@unique([assessmentId, questionId])
  @@unique([sectionId, sortOrder])
  @@index([assessmentId, sectionId])
  @@index([questionId])
}
```

If Decimal is not used, use integer hundredths and document it.

Rules:

1. Only `ACTIVE` questions may be newly assigned.
2. Same question cannot appear twice in one assessment.
3. A question belongs to one section within an assessment.
4. Points required, minimum 0.25, maximum 100.
5. Maximum 500 questions per assessment.
6. Assignment changes update `questionCount` and `totalPoints`.
7. Removing assignment does not alter question bank.
8. Archived assigned questions block publication.
9. Reorder/update is transactional.
10. Cross-category assignment is allowed by default.

## 12. Publication Readiness

Publication blockers:

1. active category exists;
2. required metadata and instructions are valid;
3. at least one section;
4. at least one assigned question;
5. every section has a question;
6. every assigned question is `ACTIVE`;
7. every assigned question remains structurally valid;
8. every assignment has valid points;
9. total points > 0;
10. aggregate question count matches;
11. slug is unique;
12. candidate-catalog assessment has compatible visibility;
13. `AVAILABLE` cannot publish with zero questions;
14. retired assessment cannot republish.

Warnings may include:

- low duration relative to question count;
- missing section instructions;
- mixed categories;
- missing question explanation/source.

Warnings do not block publication. Use stable codes and tests.

## 13. Lifecycle and Edit Policy

Supported lifecycle:

```text
DRAFT -> PUBLISHED
PUBLISHED -> ARCHIVED
ARCHIVED -> PUBLISHED
PUBLISHED -> RETIRED
ARCHIVED -> RETIRED
```

### Draft

- full metadata and structural editing
- publish when ready

### Published

Allow:

- description/instruction copy edits
- availability changes
- archive

Block:

- section changes
- assignments
- points/order
- passing score
- duration

Archive first for structural changes.

### Archived

- metadata and structural editing allowed
- readiness validation
- republish when ready

### Retired

- read-only
- never republish

Backend must enforce this.
