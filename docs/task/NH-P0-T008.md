## 2. Objective

Create one consistent repository-wide quality standard for the NextHire monorepo.

This task must establish:

- Shared ESLint configuration
- Prettier formatting rules
- Formatting and lint scripts
- Staged-file checks
- Git hooks through Husky
- Conventional Commit validation through Commitlint
- Editor recommendations
- Code-quality documentation
- Validation for the API, web app, and shared TypeScript packages

This task does not add product functionality.

---

## 3. Why This Task Is Required

NextHire will contain multiple TypeScript applications and shared packages. Without common standards, the repository can develop inconsistent formatting, conflicting lint rules, invalid imports, poor commit history, and avoidable CI failures.

The result of this task must allow developers and coding agents to use the same commands locally and later in CI.

## 9. Shared ESLint Package

Use the existing directory:

```text
packages/eslint-config
```

Set the package name:

```json
{
  "name": "@nexthire/eslint-config",
  "private": true,
  "type": "module"
}
```

Export presets equivalent to:

```json
{
  "exports": {
    "./base": "./base.mjs",
    "./node": "./node.mjs",
    "./next": "./next.mjs"
  }
}
```

### 9.1 Base Preset

The base preset must:

- Use ESLint recommended rules.
- Support TypeScript.
- Disable formatting rules that conflict with Prettier.
- Ignore generated/build directories.
- Avoid `any` in new code unless explicitly justified.
- Report unused variables.
- Allow intentionally unused arguments when prefixed with `_`.
- Prevent accidental floating promises where type-aware linting is correctly configured.
- Avoid requiring type-aware rules in files that do not have a valid TypeScript project.
- Work with ESM configuration.

Common ignored paths must include:

```text
node_modules
.next
dist
coverage
playwright-report
test-results
generated
*.min.js
```

Do not ignore source directories broadly.

### 9.2 Node/NestJS Preset

The Node preset must extend the base preset and:

- Enable Node globals.
- Support NestJS source and test files.
- Preserve necessary decorator usage.
- Avoid browser globals by default.
- Apply sensible test-file overrides.
- Preserve existing NestJS-generated rules when they are useful.

Do not introduce rules that conflict with NestJS decorators or dependency injection.

### 9.3 Next.js Preset

The Next.js preset must extend the base preset and:

- Include the installed Next.js Core Web Vitals rules.
- Include Next.js TypeScript rules when available.
- Support React Server Components.
- Allow framework-required defaults.
- Preserve accessibility-related lint rules provided by Next.js.
- Disable only rules that conflict with Prettier.

Do not disable `@next/next` rules globally.

## 15. Commitlint Configuration

Create:

```text

```

```text

```

Allowed commit types must include:

```text

fix
docs
style
refactor
perf
test
build
ci
chore
revert
```

Use conventional format:

```text
type(scope): subject
```

Valid examples:

```text
feat(auth): add email verification flow
fix(api): handle unavailable redis connection
docs(tasks): add phase zero task status
chore(repo): configure prettier and commit hooks
```

Invalid examples:

```text
updated files
New Feature
fix
```

Rules:

- Subject must not be empty.
- Header must stay within a reasonable limit such as 100 characters.
- Do not force a scope.
- Do not enforce issue IDs yet.

- No secret or environment value is printed.

- No blanket ESLint disable comments.
- No global `any` allowance.
- No disabled TypeScript strictness.
- No formatting of generated output.
