## 2. Objective

Implement this end-to-end flow:

```text
authenticated candidate opens security sessions page
        ↓
Next.js loads active sessions from NestJS
        ↓
current session is identified
        ↓
candidate can revoke one selected session
        ↓
candidate can log out from all devices
        ↓
revoked sessions cannot refresh or access protected APIs
        ↓
cookies and frontend auth state are cleared when current session is revoked
        ↓
audit events are recorded
```

The candidate must never be able to view or control another user's sessions.

## 5. Business Rules

Implement these exact rules:

1. A user can list only their own sessions.
2. A user can revoke only their own sessions.
3. Session identifiers are opaque UUIDs or repository-equivalent IDs.
4. Raw refresh tokens and token hashes are never returned.
5. Internal token-family identifiers are never returned.
6. Password hashes and private account data are never returned.
7. The current session must be clearly identified.
8. Revoking another session must not log out the current session.
9. Revoking the current session must clear the refresh cookie and frontend auth state.
10. Logout all devices must revoke every active session for the authenticated user.
11. Logout all devices must be transactional or otherwise guarantee a consistent bulk revocation result.
12. Revoked sessions cannot refresh.
13. Revoked sessions cannot continue accessing protected endpoints after session-status enforcement occurs.
14. Repeated revocation of an already revoked session is idempotent.
15. Attempting to revoke another user's session returns not-found or forbidden according to the repository's anti-enumeration convention.
16. Session list ordering defaults to most recently active first.
17. Expired sessions must not appear as active.
18. Session metadata must be privacy-minimized.
19. Raw user-agent strings should not be exposed directly if a normalized summary can be generated.
20. IP addresses must be masked or omitted from the frontend unless the privacy design explicitly approves full display.
21. Audit records must not contain refresh tokens, token hashes, cookies, authorization headers, or complete user-agent strings.
22. Logout-all audit must record the number of sessions revoked.
23. API authorization is authoritative; frontend visibility is not security.
24. All timestamps use UTC.
25. A suspended or deleted account cannot manage sessions through a stale access token.
26. Session revocation must not delete immutable audit history.

## 9. API Contracts

## 9.1 List Sessions

### Endpoint

```http
GET /api/v1/auth/sessions
```

### Access

Authenticated.

### Success

```http
200 OK
```

Logical response:

```json
{
  "sessions": [
    {
      "id": "uuid",
      "isCurrent": true,
      "status": "ACTIVE",
      "device": {
        "browser": "Chrome",
        "operatingSystem": "Linux",
        "deviceType": "desktop"
      },
      "ipAddressMasked": "192.168.x.x",
      "createdAt": "2026-07-21T10:00:00.000Z",
      "lastUsedAt": "2026-07-21T12:00:00.000Z",
      "expiresAt": "2026-08-20T10:00:00.000Z"
    }
  ]
}
```

Rules:

- query is scoped by authenticated `userId`;
- order by `lastUsedAt DESC`, then `createdAt DESC`;
- current session is determined by authenticated `sessionId`;
- expired records may be omitted by default;
- revoked records may be omitted or included only when documented;
- default recommendation: return active sessions only.

## 9.2 Revoke One Session

### Endpoint

```http
DELETE /api/v1/auth/sessions/:sessionId
```

### Access

Authenticated.

### Success

```http
204 No Content
```

Behavior:

- confirm session belongs to authenticated user;
- mark it revoked;
- set reason such as `USER_REVOKED`;
- set `revokedAt`;
- remain idempotent;
- if current session is revoked, clear refresh cookie.

Controlled errors:

```text
400 AUTH_SESSION_ID_INVALID
401 AUTH_ACCESS_TOKEN_INVALID
404 AUTH_SESSION_NOT_FOUND
```

Use anti-enumeration behavior consistent with the repository.

## 10. Session Enforcement

Inspect current access-token guard behavior.

Requirements:

1. Protected requests must validate session status.
2. A revoked or compromised session must fail.
3. An expired session must fail.
4. Session ownership must match token subject.
5. `lastUsedAt` updates must be throttled to avoid a database write on every request.

Suggested last-used update strategy:

```text
update only when existing lastUsedAt is older than 5 minutes
```

Use repository conventions and document the chosen interval.

Do not introduce an unbounded write per API request.
