## 6. Retake Business Rules

1. Only authenticated active candidates may start or access attempts.
2. Backend is authoritative for eligibility.
3. One active attempt per candidate and assessment remains enforced.
4. Finalized attempts count toward the limit.
5. First attempt is included in `maximumAttempts`.
6. Existing attempts do not change when policy changes.
7. Attempt sequence numbers are immutable and concurrency-safe.
8. Cooldown starts at the previous attempt finalization time.
9. Server time controls cooldown.
10. Unavailable, coming-soon, archived, or retired assessments cannot be retaken by default.
11. Request bodies cannot provide candidate IDs, attempt counts, or cooldown values.
12. Retake start remains transactional and idempotent.

## 7. Retake Policy

Extend assessment policy:

```ts
{
  retakeEnabled: boolean;
  maximumAttempts: number | null;
  retakeCooldownHours: number;
  certificateEnabled: boolean;
  certificateValidityDays: number | null;
}
```

Validation:

- `retakeEnabled` defaults to `false`;
- `maximumAttempts`: `null` for unlimited, otherwise 1–100;
- `retakeCooldownHours`: 0–8,760;
- `certificateEnabled` defaults to `false`;
- certificates default to supported type `CERTIFICATION`;
- `certificateValidityDays`: null or 1–3,650;
- certificate wording must not imply external accreditation.

## 12. Certificate Eligibility

Issue a certificate only when:

1. attempt is finalized and scored;
2. result is `PASSED`;
3. certificate policy is enabled;
4. assessment type is supported;
5. no certificate already exists for the attempt;
6. snapshot data is consistent;
7. a safe holder name is available.

Holder name priority:

1. valid professional/public display name;
2. verified account display name;
3. never use email as fallback;
4. fail safely if no appropriate name exists.

Certificate issuance must be idempotent.

## 14. Certificate Worker

BullMQ job:

```text
assessment.generate_certificate
```

Worker must:

1. load `PENDING`;
2. atomically change to `GENERATING`;
3. recheck eligibility and consistency;
4. generate PDF in isolated temporary storage;
5. compute SHA-256;
6. upload to private MinIO;
7. set `READY`, issue/generation dates, storage metadata, and expiry;
8. delete temporary files;
9. record safe audit event;
10. mark `FAILED` with a safe category on error;
11. use bounded retries;
12. avoid duplicate records and orphan objects.

Do not generate PDF inside the scoring transaction. Queue issuance after successful finalization commit.

## 17. Secure Download and Verification

1. Store PDFs in a private bucket.
2. Use non-guessable object keys.
3. Check candidate ownership at download time.
4. Return a short-lived signed URL, recommended 5 minutes.
5. Never store or log signed URLs.
6. Public verification uses high-entropy code.
7. Store only verification-code hash/HMAC.
8. Use strict rate limiting and anti-enumeration responses.
9. Verification page exposes limited certificate metadata.
10. Certificate pages use private/no-store caching.
11. Public verification does not reveal account/profile links by default.
