## 5. Business Rules

Implement these exact rules:

1. Only authenticated users with the `candidate` role can use this feature.
2. A candidate can access only their own education records.
3. A candidate may have multiple education records.
4. Each record has a stable opaque ID.
5. `institutionName` is required.
6. `qualification` is required.
7. `fieldOfStudy` is optional.
8. `startDate` is required.
9. `endDate` is optional when `currentlyStudying` is true.
10. `endDate` is required when `currentlyStudying` is false.
11. `endDate` cannot be earlier than `startDate`.
12. Future start dates are not allowed.
13. Future end dates are allowed only when `currentlyStudying` is true and the field is treated as expected completion; if the repository wants strict historical dates only, document and reject future end dates consistently.
14. `grade` is optional free text.
15. `description` is optional and length-limited.
16. Maximum 20 education records per candidate.
17. Duplicate records are not silently merged.
18. Ordering is candidate-controlled.
19. Reorder input must include only owned record IDs.
20. Delete is hard delete unless repository retention conventions require soft delete; document the choice.
21. Profile completion is recalculated after create, update, and delete.
22. The backend completion result is authoritative.
23. Audit metadata must not contain institution, grade, description, or other record content.
24. API authorization is authoritative.
25. Suspended or deleted accounts cannot modify records.
26. All timestamps use UTC.
27. Unexpected errors must not expose Prisma or stack traces.
28. Cross-user record access must return controlled not-found/forbidden behavior.
29. Request bodies must not accept arbitrary user IDs.
30. Empty optional strings are stored as `null` where consistent with repository conventions.

## 7. Database Design

Inspect existing Prisma conventions.

Implement the logical equivalent of:

```prisma
enum EducationLevel {
  SECONDARY
  HIGHER_SECONDARY
  DIPLOMA
  BACHELOR
  MASTER
  DOCTORATE
  PROFESSIONAL
  OTHER
}

model EducationRecord {
  id                String         @id @default(uuid()) @db.Uuid
  userId            String         @db.Uuid
  educationLevel    EducationLevel
  institutionName   String         @db.VarChar(200)
  qualification     String         @db.VarChar(150)
  fieldOfStudy      String?        @db.VarChar(150)
  startDate         DateTime       @db.Date
  endDate           DateTime?      @db.Date
  currentlyStudying Boolean        @default(false)
  grade             String?        @db.VarChar(100)
  description       String?        @db.Text
  sortOrder         Int            @default(0)
  createdAt         DateTime       @default(now())
  updatedAt         DateTime       @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, sortOrder])
  @@index([userId, startDate])
}
```

Add inverse relation on `User` when required.

### Ordering

- assign next `sortOrder` on create;
- list ascending by `sortOrder`, then newest start date;
- reorder atomically;
- normalize sort order after deletion when practical.

### Migration safety

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not run `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
- Document migration results.
