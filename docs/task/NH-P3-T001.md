```text

expert.application.manage_own
```

12. Duplicate Expert role assignment must be idempotent.
13. Approved Experts are subject to mandatory MFA policy from `NH-SEC-T001`.

## 7. Expert Application Lifecycle

Implement:

```text
DRAFT
SUBMITTED
UNDER_REVIEW
CHANGES_REQUESTED
APPROVED
REJECTED
WITHDRAWN
```

Supported transitions:

```text
DRAFT -> SUBMITTED
CHANGES_REQUESTED -> SUBMITTED
SUBMITTED -> UNDER_REVIEW
UNDER_REVIEW -> APPROVED
UNDER_REVIEW -> REJECTED
UNDER_REVIEW -> CHANGES_REQUESTED
DRAFT -> WITHDRAWN
SUBMITTED -> WITHDRAWN
CHANGES_REQUESTED -> WITHDRAWN
```

Rules:

1. New application begins as `DRAFT`.
2. Candidate may edit in `DRAFT` and `CHANGES_REQUESTED`.
3. Submitted/under-review applications are read-only for the applicant.
4. Admin begins review by moving `SUBMITTED` to `UNDER_REVIEW`, or review action may do this atomically.
5. Approval grants Expert role.
6. Rejection requires a safe reason category and reviewer note.
7. Changes requested requires a reviewer note.
8. Rejected application cannot be resubmitted in this task.
9. Withdrawal does not delete profile/document records.
10. Approved application cannot be withdrawn or edited.
11. Lifecycle enforcement must be backend-authoritative.
12. Transition helpers must be tested.

```prisma
model ExpertApplication {
  id                   String                   @id @default(uuid()) @db.Uuid

  reviewedAt           DateTime?
  reviewedById         String?                  @db.Uuid

  rejectedAt           DateTime?
  withdrawnAt          DateTime?
  createdAt            DateTime                 @default(now())
  updatedAt            DateTime                 @updatedAt

  user          User                         @relation(fields: [userId], references: [id], onDelete: Restrict)
  expertProfile ExpertProfile                @relation(fields: [expertProfileId], references: [id], onDelete: Restrict)
  documents     ExpertVerificationDocument[]

  @@index([userId, status])
  @@index([status, submittedAt])
  @@index([reviewedById, reviewedAt])

```

Enforce one active non-terminal application per user through a partial unique index or safe transactional logic.

### Verification Document

## 17. Approval Transaction

Approval service must:

1. authenticate reviewer and permission;
2. lock application;
3. confirm `UNDER_REVIEW`;
4. recheck readiness and documents;
5. verify applicant account remains active;
6. verify MFA status;
7. set `APPROVED`;
8. set reviewer/timestamps;
9. assign Expert role idempotently;
10. update required account-role metadata;
11. commit atomically;
12. record audit event;
13. return safe result.

Role-assignment failure must roll back approval.

Do not make the Expert publicly searchable yet.
