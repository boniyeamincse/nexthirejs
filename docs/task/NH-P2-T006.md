```text

        ↓
candidate sees their answer, correct answer, awarded points, and explanation
        ↓

all checks pass
```

## 6. Core Business Rules

1. Only authenticated active candidates can access result-history APIs and pages.
2. Candidate can access only their own attempts.
3. Only finalized attempts appear in history.
4. `IN_PROGRESS` attempts must not appear in result history.
5. Finalized attempts include candidate-submitted and deadline-finalized attempts.
6. Result detail is available only when scoring is complete.
7. Result history is ordered by finalization time descending, then attempt ID.
8. Source assessment edits must not alter historical result display.
9. Source question edits must not alter historical question review.
10. Review data must come from immutable attempt snapshot records.
11. Candidate answer display comes from persisted attempt-answer records.
12. Correct-answer display comes from immutable snapshot correctness data.
13. Explanations come from immutable snapshot explanation data.
14. Unanswered questions must be clearly labeled.
15. Awarded points must reconcile with stored attempt totals.
16. Candidate cannot request another candidate’s result by changing IDs.
17. Client-provided candidate/user IDs are not accepted.
18. Candidate history APIs must not expose raw internal source IDs unless necessary.
19. Audit metadata must not include candidate answers, correct answers, or explanation text.
20. Result APIs must not expose password, session, token, audit, actor, or internal storage data.
21. Pagination and filters must be validated.
22. All timestamps use UTC.
23. A blocked task must not create a Git commit.
24. A Git commit occurs only after all mandatory checks pass.

## 10. Result Outcome Rules

Determine outcome from persisted scoring data.

```text
CORRECT
INCORRECT
UNANSWERED
```

Rules:

1. No answer record or empty answer means `UNANSWERED`.
2. `isCorrect = true` means `CORRECT`.
3. `isCorrect = false` with an answer means `INCORRECT`.
4. Awarded points must match stored result.
5. Do not recalculate the attempt score during normal result reads.
6. Result read may perform consistency checks but must not silently alter historical scores.
7. If stored score data is inconsistent, return a controlled error and audit safe failure metadata.

---

## 11. Explanation Display Rules

1. Explanation is shown only after finalization.
2. Explanation comes from attempt-question snapshot.
3. Null explanation displays a neutral “No explanation provided.”
4. Explanation must follow existing sanitized text rules.
5. Never load explanation from current source question when snapshot exists.
6. Do not expose source references or internal author notes.
7. Do not expose accepted answers before finalization.

## 12. History Query Validation

Support:

```ts
{
  page?: number;
  pageSize?: number;
  search?: string;
  resultStatus?: 'PASSED' | 'FAILED';
  finalizationReason?: 'CANDIDATE_SUBMITTED' | 'DEADLINE_REACHED';
  assessmentType?: AssessmentType;
  difficulty?: AssessmentDifficulty;
  dateFrom?: string;
  dateTo?: string;
}
```

Rules:

- page minimum 1;
- page size default 12;
- page size maximum 50;
- search maximum 100 characters;
- dates use ISO date format;
- `dateFrom <= dateTo`;
- maximum date range may be limited to 5 years;
- enum filters must validate;
- deterministic ordering:
  1. submitted/finalized timestamp descending
  2. attempt ID descending

Search only snapshot title/slug or safe indexed fields.

Avoid unsafe raw SQL.

## 15. Result Consistency Checks

Before returning detailed review, verify:

1. attempt scoring is complete;
2. question count matches snapshot;
3. correct + incorrect + unanswered equals question count;
4. score earned is within valid bounds;
5. question awarded points do not exceed question possible points;
6. section totals reconcile with attempt total;
7. answer belongs to the snapshot question;
8. selected options belong to the snapshot question;
9. choice questions have valid correctness data;
10. short-text questions have accepted-answer snapshot data.

On inconsistency:

- do not silently repair;
- return controlled conflict;
- record safe failure category;
- do not expose internal values.
