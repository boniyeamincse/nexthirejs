## 9. Database Design

Inspect existing Prisma conventions.

Implement the logical equivalent of:

```prisma
enum ProfessionalLinkType {
  LINKEDIN
  GITHUB
  PORTFOLIO
  PERSONAL_WEBSITE
  BEHANCE
  DRIBBBLE
  STACK_OVERFLOW
  MEDIUM
  YOUTUBE
  OTHER
}

model CandidateAchievement {
  id           String    @id @default(uuid()) @db.Uuid
  userId       String    @db.Uuid
  title        String    @db.VarChar(200)
  issuer       String?   @db.VarChar(200)
  achievedAt   DateTime? @db.Date
  description  String?   @db.Text
  referenceUrl String?   @db.VarChar(500)
  sortOrder    Int       @default(0)
  createdAt    DateTime  @default(now())
  updatedAt    DateTime  @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, sortOrder])
  @@index([userId, achievedAt])
}

model CandidateProfessionalLink {
  id            String               @id @default(uuid()) @db.Uuid
  userId        String               @db.Uuid
  type          ProfessionalLinkType
  label         String?              @db.VarChar(100)
  url           String               @db.VarChar(500)
  normalizedUrl String               @db.VarChar(500)
  sortOrder     Int                  @default(0)
  createdAt     DateTime             @default(now())
  updatedAt     DateTime             @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, normalizedUrl])
  @@index([userId, sortOrder])
}
```

Add inverse relations on `User` when required.

Ordering:

- Assign next sort order on create.
- List ascending by sort order, then title/type.
- Reorder atomically.
- Normalize order after deletion when practical.

Migration safety:

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not run `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
