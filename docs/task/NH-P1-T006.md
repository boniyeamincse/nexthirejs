## 5. Business Rules

Implement these exact rules:

1. Only authenticated users with the `candidate` role can access this feature.
2. A candidate can access only their own profile.
3. Each user can have at most one candidate profile.
4. The profile may be created on first save and updated afterward.
5. Email remains sourced from the identity account and is not editable in this profile form.
6. Account password and authentication fields must never be included in profile responses.
7. Basic profile fields must be normalized and validated.
8. Blank optional strings should be stored as `null` when consistent with repository conventions.
9. Full name is required.
10. Professional headline is optional but length-limited.
11. Professional summary is optional but length-limited.
12. City is optional.
13. Country code is required only if the existing country configuration foundation already exists; otherwise defer country selection to `NH-P1-T007`.
14. Date of birth is optional.
15. Date of birth must not be in the future.
16. The minimum-age rule must not be hardcoded unless a country configuration rule already exists.
17. Phone number is not editable in this task unless the identity model already has a verified phone foundation.
18. Profile completion percentage must be deterministic.
19. Completion percentage must use only fields implemented in this task.
20. Completion percentage must be recalculated on every successful profile update.
21. Profile updates must be audited.
22. Audit metadata must not contain full profile content.
23. API authorization is authoritative; frontend route hiding is not security.
24. All timestamps use UTC.
25. Unexpected failures must not expose Prisma or stack-trace details.
26. Cross-user profile access must return controlled not-found/forbidden behavior.
27. Candidate-role checks must use authoritative role data.
28. Suspended or deleted accounts cannot update profiles.
