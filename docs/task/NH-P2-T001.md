## 6. Core Business Rules

1. Only authenticated active candidates can access candidate assessment catalog endpoints.
2. Candidate catalog returns only assessments with status `PUBLISHED`.
3. Draft, archived, and retired assessments must not appear in the candidate catalog.
4. A published assessment must belong to an active category.
5. Assessment title is required.
6. Assessment slug is required and globally unique.
7. Category name is required.
8. Category slug is required and globally unique.
9. Assessment type is required.
10. Difficulty level is required.
11. Visibility is required.
12. Estimated duration must be positive.
13. Maximum estimated duration is 480 minutes.
14. Question count may be zero in this foundation task because question management is not implemented yet.
15. Catalog-safe question count must come from authoritative backend data.
16. Published assessments with zero questions may appear only when explicitly marked as `COMING_SOON`.
17. Candidate detail response must not expose draft configuration, internal notes, author IDs, audit metadata, or unpublished rules.
18. Public unauthenticated assessment discovery is out of scope.
19. Pagination must be deterministic.
20. Filtering must be validated.
21. Search must be bounded and safely normalized.
22. All timestamps use UTC.
23. Client-provided user IDs are not accepted.
24. API authorization is authoritative.
25. Audit metadata must not contain profile data or future question content.
26. A blocked task must not create a Git commit.
27. A Git commit occurs only after all mandatory checks pass.

## 8. Database Design

Inspect current Prisma conventions and adapt the following logical design.

```prisma
enum AssessmentStatus {
  DRAFT
  PUBLISHED
  ARCHIVED
  RETIRED
}

enum AssessmentVisibility {
  CANDIDATE_CATALOG
  INVITE_ONLY
  INTERNAL
}

enum AssessmentDifficulty {
  BEGINNER
  INTERMEDIATE
  ADVANCED
  EXPERT
}

enum AssessmentType {
  PRACTICE
  CERTIFICATION
  SCREENING
  SKILL_CHECK
}

enum AssessmentAvailability {
  AVAILABLE
  COMING_SOON
  UNAVAILABLE
}

model AssessmentCategory {
  id          String       @id @default(uuid()) @db.Uuid
  name        String       @db.VarChar(120)
  slug        String       @unique @db.VarChar(140)
  description String?      @db.Text
  sortOrder   Int          @default(0)
  isActive    Boolean      @default(true)
  createdAt   DateTime     @default(now())
  updatedAt   DateTime     @updatedAt

  assessments Assessment[]

  @@index([isActive, sortOrder])
}

model Assessment {
  id                       String                 @id @default(uuid()) @db.Uuid
  categoryId               String                 @db.Uuid
  title                    String                 @db.VarChar(200)
  slug                     String                 @unique @db.VarChar(220)
  shortDescription         String                 @db.VarChar(500)
  description              String?                @db.Text
  instructions             String?                @db.Text
  type                     AssessmentType
  difficulty               AssessmentDifficulty
  status                   AssessmentStatus       @default(DRAFT)
  visibility               AssessmentVisibility   @default(CANDIDATE_CATALOG)
  availability             AssessmentAvailability @default(COMING_SOON)
  estimatedDurationMinutes Int
  questionCount            Int                    @default(0)
  publishedAt              DateTime?
  createdById              String?                @db.Uuid
  updatedById              String?                @db.Uuid
  createdAt                DateTime               @default(now())
  updatedAt                DateTime               @updatedAt

  category AssessmentCategory @relation(fields: [categoryId], references: [id], onDelete: Restrict)

  @@index([status, visibility, availability])
  @@index([categoryId, status])
  @@index([difficulty, type])
  @@index([publishedAt])
}
```

If creator/updater relations would cause premature role coupling, store nullable IDs only when current audit conventions support it. Document the decision.

### Constraints

Enforce in service/shared validation:

- trimmed title/name;
- normalized lowercase slug;
- duration between 1 and 480;
- question count not negative;
- `publishedAt` required when published;
- non-published records must not be candidate-visible;
- `AVAILABLE` requires published status;
- `COMING_SOON` may have zero questions;
- category must be active before publishing.

### Migration Safety

- Create a new descriptive migration.
- Do not edit applied migrations.
- Do not use `prisma migrate reset`.
- Do not use destructive `db push`.
- Generate Prisma Client.
- Validate a clean migration path.
- Document the result.
