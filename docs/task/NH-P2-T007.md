## 6. Core Business Rules

1. Only authenticated active candidates can view their performance reports.
2. Reports contain only the authenticated candidate’s finalized attempts.
3. `IN_PROGRESS` attempts are excluded.
4. Finalized deadline attempts are included unless product rules explicitly exclude them.
5. Source assessment/question edits must not alter historical report calculations.
6. Aggregation uses immutable attempt/result snapshot fields.
7. Leaderboard participation is opt-in.
8. Candidate is excluded immediately when participation is disabled.
9. Leaderboards must not expose email, phone, date of birth, location, profile-completion data, or private profile fields.
10. Display identity must use approved public profile name or generated safe alias.
11. Private candidate profiles may still participate only through safe alias when explicitly opted in.
12. Candidate can see their own rank even when outside the displayed top page, provided they are opted in.
13. Ranking must be deterministic.
14. Ties must be resolved by documented rules.
15. Candidates with no finalized attempt for a leaderboard scope are excluded.
16. One candidate contributes only their best eligible finalized attempt per assessment leaderboard.
17. Category leaderboard aggregates finalized attempts from assessments in that category.
18. Reports must not recalculate or alter stored scores.
19. Client-provided candidate IDs are not accepted.
20. Audit metadata must not include private identity or answer data.
21. All timestamps use UTC.
22. A blocked task must not create a Git commit.
23. A Git commit occurs only after all mandatory checks pass.

## 8. Percentage and Aggregation Rules

1. Use stored finalized `scorePercentage`.
2. Do not derive percentage again from source assessment data.
3. Use decimal-safe arithmetic.
4. Average is arithmetic mean of finalized attempt percentages.
5. Round display values to two decimals.
6. Pass rate:

```text
passedCount / finalizedAttemptCount * 100
```

7. Empty datasets return zero/null values according to one documented convention.
8. Historical snapshots provide assessment title/type/difficulty/category data.
9. If category snapshot is missing from prior tasks, add the minimum immutable snapshot field only if necessary and migrate safely.
10. Do not rewrite historical scores.

## 10. Assessment Leaderboard Ranking

For a specific published or historically eligible assessment:

1. Consider only opted-in active candidates.
2. Consider finalized scored attempts.
3. Use the candidate’s best attempt percentage.
4. Tie-break in this order:
   1. higher percentage;
   2. higher score earned;
   3. fewer unanswered questions;
   4. shorter valid completion duration;
   5. earlier submission timestamp;
   6. stable opaque attempt identifier.
5. Candidate appears once.
6. Default page size: 25.
7. Maximum page size: 100.
8. Rank is 1-based and stable for the same dataset.
9. Archived assessments may remain viewable through authenticated historical leaderboards if repository policy supports it; retired/internal assessments must not leak.
10. Candidate catalog visibility rules remain authoritative.

---

## 11. Category Leaderboard Ranking

For a category:

1. Consider opted-in active candidates.
2. Use finalized attempts from assessments in the category.
3. Minimum eligible unique assessments: 1.
4. Aggregate candidate metrics:

```text
averagePercentage
bestPercentage
completedAssessmentCount
passedAssessmentCount
passRate
```

5. Rank by:
   1. higher average percentage;
   2. higher completed unique assessment count;
   3. higher pass rate;
   4. higher best percentage;
   5. earlier latest qualifying completion;
   6. stable opaque candidate ranking key.
6. One best attempt per candidate per assessment contributes.
7. Do not let repeated attempts inflate assessment count.
