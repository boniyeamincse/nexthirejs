## 6. Business Rules

1. Only authenticated active candidates may request exports or deactivate their account.
2. A candidate can access only their own export requests and files.
3. Password confirmation is required for deactivation.
4. Wrong password returns a generic controlled error.
5. Deactivation sets account status to the existing inactive/deactivated state; do not invent a conflicting status.
6. Successful deactivation revokes every active session, including the current session.
7. Successful deactivation disables external profile discoverability and link-only sharing.
8. A deactivated account cannot log in, refresh tokens, access protected APIs, or appear publicly.
9. Deactivation is not permanent deletion.
10. Existing database records remain according to retention policy.
11. At most one active export request may exist per candidate.
12. Export generation is asynchronous.
13. Export request statuses are deterministic.
14. The export archive must contain only the candidate's own data.
15. Authentication secrets, password hashes, refresh-token hashes, raw tokens, internal audit implementation data, and system secrets must never be exported.
16. Export files must have limited retention.
17. Recommended export expiry: 7 days.
18. Download authorization must be checked at download time.
19. Download URLs/tokens must be short-lived and unguessable.
20. Raw download tokens must not be stored or logged.
21. Export archive must not be emailed as an attachment.
22. Export file content must be structured and understandable.
23. Recommended archive format: ZIP containing JSON files plus a README.
24. Export generation failures must be safely reported without stack traces.
25. Rate limiting applies to export requests and deactivation attempts.
26. Audit metadata must not contain passwords, export file contents, raw URLs, or tokens.
27. All timestamps use UTC.
28. A blocked task must not create a Git commit.
29. A Git commit occurs only after all mandatory checks pass.

## 13. Export Worker

Suggested BullMQ job:

```text
candidate.generate_data_export
```

Worker responsibilities:

1. Validate request is still `PENDING`.
2. Transition atomically to `PROCESSING`.
3. Load candidate-owned data.
4. Map data to export-safe structures.
5. Generate ZIP in temporary storage.
6. Calculate SHA-256 checksum.
7. Upload to private object storage.
8. Set `READY`, size, checksum, completion, and expiry.
9. Delete temporary files.
10. Record audit event.
11. On failure, set `FAILED` with safe category.
12. Never log archive content or sensitive fields.

Use bounded retries and idempotency. A retry must not create duplicate active files.
