## 6. Core Business Rules

1. Only authenticated active candidates can start or access attempts.
2. Candidate role is enforced by the API.
3. Candidate can access only their own attempts.
4. An attempt may be created only from a `PUBLISHED` assessment.
5. The assessment must be `AVAILABLE`.
6. The assessment must be visible in the candidate catalog.
7. The assessment category must remain active at start time.
8. Publication readiness must still be valid at start time.
9. Exactly one editable active attempt may exist per candidate and assessment.
10. Repeated start requests must return the existing active attempt instead of creating duplicates.
11. Attempt creation and snapshot generation must be transactional.
12. Attempt snapshot is immutable after creation.
13. Later edits or republishing of the source assessment must not alter an existing attempt.
14. Correct answers, accepted answers, explanations, and scoring keys must never be included in candidate API responses.
15. Candidate answer drafts are stored separately from snapshot correctness data.
16. Server time is authoritative.
17. `deadlineAt` is calculated from `startedAt + estimatedDurationMinutes`.
18. Client countdown is informational; backend enforces deadline.
19. Answer writes after deadline are rejected.
20. An overdue active attempt becomes `EXPIRED`.
21. Expired attempts are read-only.
22. Final submission is not implemented in this task.
23. Draft answers may be created, updated, or cleared before the deadline.
24. Answer payload must match question type.
25. Progress is backend-calculated.
26. Reordering or source-question updates cannot change an attempt snapshot.
27. Request bodies must not accept candidate/user IDs.
28. Raw correctness data must not appear in logs or audit metadata.
29. All timestamps use UTC.
30. A blocked task must not create a Git commit.
31. A Git commit occurs only after all mandatory checks pass.

2. Re-run readiness validation.
3. Confirm publication version and availability.

6. Copy ordered sections.
7. Copy ordered assignments and question content.
8. Copy options and correctness data.
9. Copy accepted answers and explanation for future scoring/result use.
10. Verify copied question count and points match assessment aggregates.
11. Calculate `deadlineAt` from database/server time.
12. Commit.
13. Return candidate-safe attempt DTO.

If readiness or aggregate reconciliation fails, do not create a partial attempt.

## 16. Deadline and Expiration Enforcement

Create one server-side attempt-state service.

Rules:

1. On every attempt read/write, compare server time with `deadlineAt`.
2. If overdue and status is `IN_PROGRESS`, transition atomically to `EXPIRED`.
3. Expired workspace may be returned read-only.
4. Save/clear answer operations after expiry return controlled error.
5. Frontend countdown never extends the deadline.
6. Client clock manipulation has no authority.
7. `lastActivityAt` updates should be throttled to avoid excessive writes.
8. Expiration logic must be race-safe.
9. Automatic final submission is out of scope.
10. A later scheduled expiry worker may be added in a future task; lazy expiration is mandatory here.
