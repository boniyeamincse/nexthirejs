## 5. Business Rules

1. Only authenticated `candidate` users may access this dashboard.
2. A candidate sees only their own completion data.
3. Backend completion is authoritative.
4. Frontend must not calculate the official percentage independently.
5. Use the latest versioned completion score.
6. Overall percentage must remain between 0 and 100.
7. Every implemented profile section must appear.
8. Each section status is `NOT_STARTED`, `IN_PROGRESS`, or `COMPLETED`.
9. Section points must reconcile exactly with the overall score.
10. Missing actions must be deterministic and based only on implemented fields.
11. Action links must target existing routes.
12. Privacy visibility does not change completion.
13. Hidden sections still contribute to completion.
14. Reordering records does not alter completion.
15. Suspended or deleted accounts cannot access the dashboard.
16. Responses must omit passwords, sessions, tokens, audit data, and raw private profile values.
17. Audit metadata must not contain profile values.
18. User IDs must not be accepted from path, query, or body.
19. All timestamps use UTC.
20. A blocked task must not create a Git commit.

## 7. Section Status Rules

Create one reusable backend section-status service.

### `NOT_STARTED`

No meaningful valid data exists.

### `IN_PROGRESS`

Some valid data exists, but the section has not earned all configured points.

### `COMPLETED`

The section has earned all configured points.

Requirements:

- reuse the same source data and scoring rules as the authoritative completion service;
- avoid duplicating weight logic;
- expose earned and possible points per section;
- section possible points must total 100;
- section earned points must total the overall earned points;
- status calculation must be deterministic;
- the dashboard version must match the completion version.
