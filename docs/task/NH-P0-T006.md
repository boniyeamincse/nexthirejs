- Avoid duplicate configuration logic.

## 10. Redis Service

Create a singleton Redis service using `ioredis`.

Required public methods:

```typescript
ping(): Promise<string>;
getClient(): Redis;
isReady(): boolean;
disconnect(): Promise<void>;
```

Behavior:

- Connect during module initialization.
- Close cleanly during module destruction.
- Use NestJS `Logger` for safe connection-state messages.
- Throw a clear error if Redis cannot connect.
- Never log credentials.
- Do not create application cache methods yet.
- Avoid open Jest handles.

Use a dedicated injection token for the Redis client if that better matches the project style.

## 11. BullMQ Configuration

Register BullMQ through a dedicated queue module.

Requirements:

- Reuse Redis connection settings.
- Prefix from `BULLMQ_PREFIX`.
- Default attempts from the environment.
- Exponential backoff using the configured delay.
- Configured completed-job retention.
- Configured failed-job retention.
- No business queues.
- No queue dashboard.

Create constants:

```typescript
export const SYSTEM_HEALTH_QUEUE = 'system-health';
export const SYSTEM_HEALTH_PING_JOB = 'ping';
```

Do not scatter queue or job string literals throughout the codebase.

## 12. Queue Producer Service

Create a queue service with:

```typescript
enqueuePing(input: {
  requestedAt: string;
  source: string;
}): Promise<{
  jobId: string | number | undefined;
  queue: string;
  name: string;
}>;
```

Requirements:

- Use `@InjectQueue`.
- Add only the `ping` job to the `system-health` queue.
- Return safe job metadata only.
- Do not accept arbitrary queue names or job names.
- Do not return Redis details.

## 13. Sample Processor

Create a BullMQ processor for `system-health`.

Required behavior:

- Process the `ping` job.
- Validate the payload defensively.
- Throw on malformed payload.
- Do not call the database or external APIs.
- Return:

```json
{
  "status": "processed",
  "source": "manual-test",
  "processedAt": "ISO_TIMESTAMP"
}
```

Use BullMQ retry behavior rather than custom retry loops.

## 14. Development-Only Test Endpoint

Create:

```text
POST /api/v1/system/queue/ping
```

Request:

```json
{
  "source": "manual-test"
}
```

DTO requirements:

- `source` is required.
- String only.
- Trim whitespace.
- Minimum length 2.
- Maximum length 50.

Response status:

```text
202 Accepted
```

Response body:

```json
{
  "status": "queued",
  "queue": "system-health",
  "job": "ping",
  "jobId": "123"
}
```

Feature-control requirements:

- Endpoint works only when `SYSTEM_QUEUE_TEST_ENABLED=true`.
- When disabled, return 404 or 403 consistently with the project error style.
- Do not accept arbitrary job payloads.
- Do not expose Redis configuration.
- Document that this endpoint is for local/infrastructure verification and must not remain publicly enabled in production.

Do not add authentication in this task.
