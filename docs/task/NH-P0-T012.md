## 2. Objective

Create stable module boundaries and reusable infrastructure for future identity features:

```text
HTTP request
  ↓
correlation/request context
  ↓
authentication boundary
  ↓
future authorization boundary
  ↓
business service
  ↓
audit service
  ↓
immutable audit record
```

After this task:

- `AuthModule` exists as the single backend authentication boundary.
- `AuditModule` exists as the single backend audit-writing boundary.
- Every API request has a correlation ID.
- Future controllers can mark public routes explicitly.
- Future authenticated requests have a typed request-principal contract.
- Sensitive operations can write structured audit records through one service.
- Audit metadata is sanitized and bounded.
- Audit logs cannot be updated through the application service.
- Swagger and documentation explain the foundation.
- Tests verify request context and audit behavior.

No real authentication session is created in this task.

## 6. Correlation and Request Context

### 6.1 Header

Use:

```text
X-Request-Id
```

Requirements:

1. Accept a valid client-provided request ID only if it passes strict validation.
2. Otherwise generate a new UUID.
3. Attach the value to the request context.
4. Return it in the response header.
5. Make it available to application services and the audit service.
6. Include it in structured application logs when the existing logger supports contextual metadata.
7. Do not trust arbitrary oversized or malformed header values.

Recommended validation:

- UUID format, or
- repository-approved restricted token format
- maximum 100 characters

Prefer UUID generation when no valid value is supplied.

### 6.2 Request Context Contract

Create a typed request-context contract containing only what is currently available:

```ts
interface RequestContext {
  requestId: string;
  actor?: AuthenticatedPrincipal;
  ipAddress?: string;
  userAgent?: string;
}
```

Security rules:

- Do not persist raw IP address or user agent automatically in every audit record.
- Do not expose proxy headers without respecting the application's trusted-proxy configuration.
- Do not create fake authenticated principals.
- Anonymous requests must remain anonymous.

Use the existing framework pattern if AsyncLocalStorage or another request-context solution already exists. Do not add a second competing context system.


### 7.2 Authenticated Principal Contract

Create a minimal typed contract:

```ts
interface AuthenticatedPrincipal {
  userId: string;
  sessionId?: string;
  roleCodes: string[];
  permissionCodes: string[];
```

Rules:

- This is a request identity contract, not a database entity.
- Do not include password, token, refresh token, email-verification token, or private profile fields.
- Do not attach a principal to requests until a future authentication mechanism validates one.
- Add focused unit tests for metadata and type-facing helpers where meaningful.

### 7.3 Naming

Use stable names that future guards, decorators, and audit services can consume. Avoid names tied only to candidates because the platform also supports trainers, companies, and internal users.

## 8. Audit Data Model

Inspect existing Prisma naming and ID conventions before implementation.

Create a logical model equivalent to:

```prisma
enum AuditActorType {
  ANONYMOUS
  USER
  SYSTEM
  INTERNAL
}

enum AuditOutcome {
  SUCCESS
  FAILURE
  DENIED
}

model AuditLog {
  id           String         @id @default(uuid()) @db.Uuid
  occurredAt   DateTime       @default(now())
  requestId    String?        @db.VarChar(100)
  actorType    AuditActorType
  actorUserId  String?        @db.Uuid
  action       String         @db.VarChar(150)
  targetType   String?        @db.VarChar(100)
  targetId     String?        @db.VarChar(100)
  outcome      AuditOutcome   @default(SUCCESS)
  metadata     Json?
  createdAt    DateTime       @default(now())

  @@index([occurredAt])
  @@index([actorUserId, occurredAt])
  @@index([action, occurredAt])
  @@index([targetType, targetId])
  @@index([requestId])
}
```

Adapt field mapping and IDs to current repository conventions.

### 8.1 Immutability

Application rules:

- Provide create/record operations only.
- Do not add update or delete methods.
- Do not expose CRUD endpoints.
- Do not use `updatedAt`.
- Do not cascade-delete audit records when a future user is deleted.
- `actorUserId` remains a scalar reference in this foundation unless the current schema has a safe non-destructive relation pattern.

### 8.2 Migration Safety

- Create a new descriptive migration.
- Never edit an already-applied migration.
- Never run `prisma migrate reset`.
- Never use destructive `db push`.
- Run Prisma formatting, validation, migration, and generation through repository scripts.
- Document the migration name and result.

## 9. Audit Service

Create a single audit-writing service with a contract similar to:

```ts
interface RecordAuditEventInput {
  action: string;
  actorType: AuditActorType;
  actorUserId?: string;
  targetType?: string;
  targetId?: string;
  outcome?: AuditOutcome;
  metadata?: Record<string, unknown>;
  requestId?: string;
}
```

### 9.1 Required Behavior

The service must:

1. Validate the action name.
2. Use request context for `requestId` when not explicitly supplied.
3. Default outcome to `SUCCESS`.
4. Store only JSON-safe sanitized metadata.
5. Reject or remove sensitive keys.
6. enforce metadata size/depth limits.
7. Never throw raw Prisma details to callers.
8. Use the existing application logger for internal failures.
9. Keep audit-write behavior deterministic and testable.

### 9.2 Action Naming Convention

Use dot-separated action names:

```text
domain.resource.action
```

Examples for future tasks:

```text
auth.candidate.registered
auth.email_verification.requested
company.job.published
finance.refund.approved
```

Validation requirements:

- lowercase letters
- numbers where needed
- dots and underscores only
- maximum 150 characters
- no whitespace

### 9.3 Sensitive Metadata Filtering

At minimum, reject or redact case-insensitive keys matching:

```text
password
passwordHash
token
accessToken
refreshToken
authorization
cookie
secret
otp
verificationCode
privateKey
cardNumber
cvv
```

Also prevent:

- functions
- symbols
- BigInt serialization errors
- circular references
- excessive object depth
- excessive array length
- oversized strings
- oversized total JSON payload

Use sensible documented limits, for example:

```text
maximum depth: 5
maximum metadata JSON size: 16 KB
maximum string length: 2,000 characters
maximum array length: 100 items
```

Tests must prove sensitive values are not persisted.

### 9.4 Failure Strategy

For this foundation, choose and document one strategy:

- **strict:** the business operation fails if mandatory audit writing fails, or
- **best effort:** the business operation continues but audit failure is logged and monitored.

The default recommendation for this generic service is:

- expose both `recordRequired()` and `recordBestEffort()` methods, or
- one method with an explicit mode

Do not silently choose best effort for finance/admin/security actions. Future feature tasks must select the mode according to business risk.

## 12. Tests

### 12.1 Request Context Tests

Test:

1. A missing request ID generates a UUID.
2. A valid request ID is preserved.
3. An invalid request ID is replaced.
4. An oversized request ID is replaced.
5. The response contains `X-Request-Id`.
6. Concurrent requests do not share context.
7. Anonymous requests contain no authenticated principal.

### 12.2 Auth Foundation Tests

Test:

1. `@Public()` writes the expected metadata.
2. The metadata key is stable and exported from one location.
3. Principal helpers never infer or fabricate authentication.
4. No global authentication guard blocks existing endpoints.

### 12.3 Audit Sanitization Tests

Test:

1. A valid event is accepted.
2. Invalid action naming is rejected.
3. Sensitive keys are redacted or removed.
4. Sensitive keys are detected case-insensitively.
5. Nested sensitive keys are filtered.
6. Circular metadata does not crash the process.
7. BigInt metadata is handled safely.
8. Excessive depth is rejected or truncated according to the documented rule.
9. Oversized metadata is rejected.
10. Undefined/function/symbol values are not persisted.

### 12.4 Audit Service Unit Tests

Test:

1. Request ID is sourced from request context.
2. Explicit request ID takes precedence when allowed.
3. Outcome defaults to `SUCCESS`.
4. Anonymous actor requires no user ID.
5. USER actor validates user ID format when UUIDs are used.
6. Safe metadata reaches the repository.
7. Prisma errors are converted to safe application errors.
8. Best-effort mode logs failures and does not expose Prisma details.
9. Required mode reports a controlled failure.

### 12.5 Database Integration Tests

Using the local/test PostgreSQL database:

1. Create an anonymous audit record.
2. Create a user-actor audit record without requiring a user relation.
3. Query by request ID.
4. Query by action.
5. Verify timestamps are UTC-compatible.
6. Verify metadata is valid JSON.
7. Confirm no application update/delete repository methods exist.
8. Confirm migration applies cleanly.

### 12.6 Regression Tests

Run all existing API, web, shared-package, and root tests to ensure the foundation does not break current behavior.
