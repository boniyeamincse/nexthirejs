## 1. AI Development Instruction

Read this task completely, inspect the repository, and implement only `NH-P0-T011`.

The purpose of this task is to add a reliable continuous integration pipeline for the NextHire pnpm and Turborepo monorepo. The pipeline must validate the Next.js frontend, NestJS API, Prisma schema, and all existing shared packages on every pull request and on pushes to protected development branches.

Before changing files:

1. Inspect the complete repository.
2. Read `README.md`.
3. Read `docs/development/code-quality.md`.
4. Read `docs/task/status.md`.
5. Inspect root and workspace `package.json` scripts.
6. Inspect `pnpm-workspace.yaml`, `turbo.json`, lockfile, Node version files, existing Docker configuration, and any existing CI workflows.
7. Determine the actual branch names used by the repository.
8. Run the existing local quality commands to identify the baseline.

Do not initialize new applications, implement product features, modify business logic, redesign the web UI, add authentication, or change the database schema in this task.

If an equivalent complete CI workflow already exists, do not create a duplicate workflow. Verify it against this task, improve only missing requirements, document the result, update `docs/task/status.md`, and stop.

## 2. Objective

Create a production-quality CI workflow that automatically verifies:

- Dependency installation using the committed pnpm lockfile
- Formatting
- ESLint
- TypeScript type checking
- Unit tests
- API integration or E2E tests when currently available
- Web E2E tests when currently available and CI-ready
- Prisma formatting and schema validation
- NestJS API build
- Next.js web build
- Shared-package builds
- Secret-safe environment handling
- Docker Compose configuration validity when supported by the existing setup

The pipeline must fail when any mandatory quality gate fails.

## 4. Scope

Implement only the following:

### 4.1 CI Workflow

Create or update the repository CI workflow under the platform already used by the repository.

For GitHub repositories, the expected location is:

```text
.github/workflows/ci.yml
```

The workflow must run on:

- Pull requests targeting the main development branches
- Pushes to the main development branches
- Manual execution through `workflow_dispatch`

Use the actual branch names discovered in the repository. Do not invent a branch without documenting the decision.

### 4.2 Runtime Setup

The workflow must:

1. Check out the repository.
2. Configure Node.js 22.
3. enable Corepack.
4. activate or prepare the pnpm version required by the root `packageManager` field.
5. cache pnpm's package store.
6. install dependencies with the frozen lockfile.
7. avoid globally installing an unrelated pnpm version.

Expected installation behavior:

```bash
corepack enable
pnpm install --frozen-lockfile
```

### 4.3 Quality Gates

Run repository-defined equivalents of:

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

Do not replace existing scripts with direct tool calls unless a script is missing and adding one is within this task.

### 4.4 Prisma Verification

Verify the Prisma schema without applying migrations to a production or unknown database.

Run repository-defined equivalents of:

```bash
pnpm --filter @nexthire/api prisma:format
pnpm --filter @nexthire/api prisma:validate
pnpm --filter @nexthire/api prisma:generate
```

When script names differ, use the actual scripts and document them.

CI must not run destructive commands such as:

```bash
prisma migrate reset
prisma db push --accept-data-loss
```

### 4.5 Test Infrastructure

Inspect the current tests before deciding whether service containers are required.

When API integration or E2E tests require PostgreSQL or Redis:

- Add PostgreSQL and Redis CI services.
- Use health checks.
- Use non-production credentials.
- Pass configuration only through CI environment variables.
- Run migrations through the repository's safe CI migration command.
- Do not require MinIO or Mailpit unless an existing mandatory test actually depends on them.

When no current test requires database or Redis services:

- Do not add unnecessary services.
- Still run Prisma formatting, validation, and client generation.
- Document that database-backed tests will be enabled in their feature task.

### 4.6 E2E Tests

Run existing E2E suites only when they are already implemented and stable.

For example:

```bash
pnpm --filter @nexthire/api test:e2e
pnpm --filter @nexthire/web test:e2e
```

If an E2E command exists but requires missing fixtures or an unfinished application feature:

- Do not hide the failure using `continue-on-error`.
- Either make the existing foundation test CI-safe without expanding feature scope, or document it as a blocker.
- Do not create full application E2E tests in this CI task.

### 4.7 Docker Configuration Validation

If Docker Compose is part of the repository and available in the CI runner, validate its configuration:

```bash
docker compose config --quiet
```

Use the actual Compose file path.

Do not start all infrastructure containers unless required by mandatory tests.

### 4.8 Concurrency and Permissions

Configure CI with least privilege.

Recommended GitHub Actions permissions:

```yaml
permissions:
  contents: read
```

Add concurrency so a newer run for the same branch can cancel an obsolete in-progress run.

Do not grant write permissions, package publishing access, deployment access, or cloud credentials.

### 4.9 Documentation and Status

Update:

```text
README.md
docs/development/code-quality.md
docs/task/status.md
```

Document:

- What triggers CI
- Mandatory quality gates
- Required Node and pnpm versions
- How developers run the same checks locally
- Whether database and Redis services are used
- How to diagnose a failed workflow
- The exact workflow file path

## 6. Workflow Design Requirements

Prefer a clear workflow with either:

### Option A — Single Quality Job

Use one job when repository checks are fast and simple.

Suggested sequence:

```text
checkout
→ setup Node/Corepack/pnpm
→ install
→ Prisma validation/generation
→ format check
→ lint
→ typecheck
→ tests
→ E2E tests when available
→ build
→ Docker Compose validation
```

### Option B — Multiple Jobs

Use multiple jobs when meaningful parallelism already exists.

Possible jobs:

```text
quality
api-tests
web-tests
build
```

Requirements for multiple jobs:

- Do not repeat expensive work without a reason.
- Preserve easy-to-read failure reporting.
- Do not create a complex matrix for only two applications.
- All mandatory jobs must be required for workflow success.
- Keep Node and pnpm versions consistent.

Choose the simplest design that fits the current repository.

## 7. Environment Safety

Use only safe CI values.

Example local-only CI variables may include:

```text
NODE_ENV=test
DATABASE_URL=postgresql://nexthire:nexthire_test@localhost:5432/nexthire_test
REDIS_URL=redis://localhost:6379
NEXT_PUBLIC_API_BASE_URL=http://localhost:3001/api/v1
```

Adapt names to the existing `.env.example`.

Rules:

1. Never copy real secrets into the workflow.
2. Never print database passwords, tokens, or connection strings unnecessarily.
3. Do not commit `.env`, `.env.local`, or production credentials.
4. Prefer test defaults for non-sensitive values.
5. Use repository secrets only when an existing mandatory test requires a real external service; otherwise mock or skip that external integration according to existing test architecture.
6. CI builds must not require live payment, email, SMS, video, AI, or object-storage providers.

Examples:

```json

}
```

Do not replace Turborepo orchestration with duplicated manual commands unless necessary.

### Step 6 — Validate the Workflow Locally

Run all commands used by CI locally.

Run:

```bash
git diff --check
docker compose config --quiet
```

when applicable.

A local command passing does not guarantee YAML correctness. Review workflow syntax and action inputs carefully.

### Step 7 — Update Documentation
